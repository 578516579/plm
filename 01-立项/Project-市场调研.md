# Project 模块 - 市场调研（内部工具语境简化版）

> ⚠️ Project 模块是 **PLM 内部研发管理工具的核心实体**，不是面向外部市场销售的产品。本"市场调研"按 Gate Checklist 要求填写，但目的是 **验证团队内部需求 + 调研类似工具的实现思路**，不做 TAM/SAM/SOM 量化。

## 1. 调研背景与目标

回答 4 个问题：
1. 团队内部有没有真实痛点？（vs 飞书表格够不够用）
2. 类似工具（开源 / SaaS）能不能直接用，省得自己造轮子？
3. 如果自己做，哪些字段 / 功能是 must-have？
4. 业务上能否承载 PLM 后续 Phase / Task / Requirement 子实体？

---

## 2. 内部用户访谈（简化为 3 段）

> 访谈对象：研发团队 PM、技术 lead、运营负责人。每段 5-10 分钟。

### 2.1 PM 老王
> "现在用飞书表格还行，但有 2 个问题——一是状态字段经常忘改、跟周报对不上；二是一个项目要关联哪些任务，要么手动维护链接、要么开飞书任务的另一个表。如果有强外键的话很解决。"

→ **关键诉求**：状态字段必填 + 强校验、能关联子实体。

### 2.2 技术 lead
> "我关心的不是 UI，是数据。能不能让我 SQL 直接查到'本季度研发类项目的平均预算 / 状态分布'？还有项目编号要规范，飞书表格那种'PRJ-2024-001'手动维护经常重号。"

→ **关键诉求**：结构化字段 + 项目编号自动化。

### 2.3 运营负责人
> "我们运营周报要列项目状态，现在每周花 20 分钟跟 PM 对账。如果有个能导出 Excel 的就好了。"

→ **关键诉求**：导出能力 + 数据准确性。

**共同结论**：自建一个轻量级的项目实体管理足够，**核心是字段规范 + 状态强校验 + 导出**，UI 不用花哨。

## 3. 类似工具对比

| 工具 | 优势 | 劣势 | 是否可直接用？ |
|---|---|---|---|
| **飞书多维表格** | 上手快 / 已有 | 字段规范靠人 / 无 API 集成 / 无外键 | 当前在用，但触顶 |
| **禅道（开源）** | 全套项目/Bug/任务/迭代 | 偏 IT 项目管理，UI 老旧 / 自部署沉重 | 牛刀杀鸡 |
| **Jira** | 业界标准 | 收费 / 国内访问慢 / 学习曲线陡 | 成本不划算 |
| **PingCode / 蓝湖等 SaaS** | 国内体验好 | 收费、数据要出公司 / 难定制 | 数据合规上不可行 |
| **自建（在 PLM 内）** | 与若依集成 / 完全可定制 / 数据自主 | 要造轮子（但 ruoyi-bootstrap skill 半自动化） | ✅ 选定方案 |

## 4. 结论与建议

**结论**：**自建**，优先级 P0。

**理由**：
1. 现有工具或贵或重，且数据要在自己手里
2. PLM 系统已经基础设施就位（用户/角色/菜单/字典），加一张 `tb_project` 表 + 简单 CRUD 边际成本极低（ruoyi-bootstrap skill Phase 7 几乎一键完成）
3. 后续 Phase / Task / Requirement / Change 等子实体都要 FK 到 `tb_project`，Project 是 PLM 业务模型的根

**建议范围**（→ PRD 第 1 期）：
- 必做：CRUD + 字段规范（编号唯一、状态强校验、负责人 FK）+ 搜索 + 导出
- 不做：模板复制、甘特图、项目级权限隔离 → v0.2

**风险**：
- ⚠️ 项目编号规则未定 → Phase 02 必须决
- ⚠️ 标签功能模型未定 → 写进 PRD 但延到 v0.2
