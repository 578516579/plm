# ADR-0001: Project 模块的 `project_no` 编号规则

- **状态**：accepted
- **日期**：2026-05-15
- **决策人**：Wjl（技术 lead + 产品 Owner，solo-review）
- **关联**：[Project-PRD.md §6 Q1](../../01-立项/Project-PRD.md) / [Project-数据库设计.md](../../02-设计/Project-数据库设计.md)

## 背景

PRD 立项时把 `project_no` 规则列为 Q1 开放问题，留 Phase 02 决。3 个候选方案：

| 候选 | 例子 | 优点 | 缺点 |
|---|---|---|---|
| **A** DB 自增 ID 直显 | `1`, `2`, `3`... | 永不重号、零代码 | 缺业务感、不便人工沟通（"项目 1734 进度怎样？"很怪） |
| **B** 前缀 + 年份 + 流水号 | `PRJ-2026-0001` | 人类友好、便于邮件 / 飞书提及；可看出立项年份 | 流水号管理：DB 序列 vs 全局表 vs Service 层算 |
| **C** 业务自定义 | 用户自填，规则千奇百怪 | 完全灵活 | 容易重号；统计 / 排序困难 |

## 决策

**选 B：`PRJ-YYYY-NNNN`**，4 位流水号按年滚动。

具体规则：
- `PRJ` 前缀固定
- `YYYY` 立项当年（由后端 fill，不允许人工改）
- `NNNN` 当年内流水号，0001 起，**Service 层在 INSERT 前查"当年最大流水号 + 1"**，加 DB UNIQUE 索引兜底
- 满 9999 → 升 5 位（`NNNNN`），向后兼容

**对比反方案**：
- 不选 A：缺业务感，邮件 / 飞书提及项目时不友好
- 不选 C：v0.1 阶段团队小，DIY 编号容易混乱；后续如需可加"自定义编号"字段（不替换 `project_no` 而是平行字段）

## 理由

- 与传统 PM 习惯一致（飞书表格 / Excel 时代大家都用 PRJ-2024-XXX 这类）
- `YYYY` 让"按年聚合"看一眼就懂
- DB UNIQUE 索引 + Service 层 try-catch DuplicateKeyException 双重防护
- 当年内"最大流水号 + 1" 在并发新建时可能撞同号，靠 DB 索引兜底 + Service 层重试 1 次即可

## 后果

### 好

- 编号规范、可读、可统计
- 与团队既有习惯无缝衔接
- 实现成本极低（Service 层 1 个方法）

### 代价

- Service 层每次 insert 多 1 次 SELECT 查最大流水号
- 跨年时（12-31 23:59 vs 01-01 00:00）边界要测试一下
- 流水号断号（中间删项目）不补；这是可接受的（按 PRD del_flag 逻辑删除，记录仍在）
