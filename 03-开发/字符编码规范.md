# 字符编码规范（UTF-8 全栈）

| 字段 | 值 |
|---|---|
| 版本 | v1.0 |
| 触发原因 | 2026-05-16 三模块 E2E 数据库出现 `EFBFBD` 替换字符（mojibake）；根因为 Windows JDK 默认 `file.encoding=GBK` |
| 适用范围 | 全部 plm-* 后端模块 + plm-frontend + sql/ + .claude/ 工具链 |
| 强制级别 | **MUST**（出现任何一处违反即视为 P0 缺陷） |

---

## 1. 黄金法则

```
全栈 UTF-8 · 无 BOM · LF 换行 · 任何一环错位都会出现 EFBFBD
```

链路图（任一处错就乱）：

```
源码文件 (UTF-8/无BOM)
    ↓ javac -encoding UTF-8
.class (UTF-16 BMP)
    ↓ java -Dfile.encoding=UTF-8 (必须!)
JVM 内 String (UTF-16)
    ↓ Tomcat connector URIEncoding=UTF-8
HTTP 请求/响应 (Content-Type: charset=UTF-8)
    ↓ MySQL connector characterEncoding=utf8
JDBC ResultSet
    ↓ MySQL session character_set_* = utf8mb4
表/列 COLLATE utf8mb4_0900_ai_ci
```

## 2. 配置清单（10 处必查）

### 2.1 JVM 启动参数（最容易遗漏）

```bash
java -Dfile.encoding=UTF-8 \
     -Dsun.jnu.encoding=UTF-8 \
     -jar plm-admin/target/plm-admin.jar
```

或在 IDE / `pom.xml` 编译插件：

```xml
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-compiler-plugin</artifactId>
  <configuration>
    <encoding>UTF-8</encoding>
  </configuration>
</plugin>
```

**陷阱**：Windows 中文 JDK 默认 `file.encoding=GBK`，必须显式覆盖。

### 2.2 Spring Boot `application.yml`

```yaml
server:
  servlet:
    encoding:
      charset: UTF-8
      enabled: true
      force: true   # 关键!
  tomcat:
    uri-encoding: UTF-8
```

### 2.3 JDBC URL

```yaml
spring:
  datasource:
    url: jdbc:mysql://host:3306/plm?useUnicode=true&characterEncoding=utf8&serverTimezone=GMT%2B8
```

### 2.4 MySQL 服务端（一次性）

```ini
# my.cnf / my.ini
[mysqld]
character-set-server=utf8mb4
collation-server=utf8mb4_0900_ai_ci
init-connect='SET NAMES utf8mb4'

[client]
default-character-set=utf8mb4
```

### 2.5 MySQL 命令行导入

```bash
mysql --default-character-set=utf8mb4 -u root -p plm < sql/business-x.sql
# ⚠️ 漏掉 --default-character-set 会被 Windows GBK 客户端污染！
```

### 2.6 建表 DDL

```sql
CREATE TABLE tb_x (...) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 2.7 Maven `pom.xml`

```xml
<properties>
  <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
  <project.reporting.outputEncoding>UTF-8</project.reporting.outputEncoding>
</properties>
```

### 2.8 `.editorconfig`（编辑器层）

```ini
root = true
[*]
charset = utf-8
end_of_line = lf
insert_final_newline = true
```

### 2.9 Git `.gitattributes`

```
* text=auto eol=lf
*.sh   text eol=lf
*.bat  text eol=crlf
*.java text eol=lf working-tree-encoding=UTF-8
```

### 2.10 PowerShell / cmd 控制台（Windows）

```powershell
# PowerShell 显式 UTF-8
$OutputEncoding = [Console]::OutputEncoding = [System.Text.UTF8Encoding]::new()
chcp 65001
```

## 3. 自检脚本（commit 前必跑）

放在 `plm-backend/scripts/check-encoding.sh`（见同名文件）：

```bash
#!/usr/bin/env bash
set -e

# 1. 源码文件必须 UTF-8 无 BOM
find . -name "*.java" -o -name "*.xml" -o -name "*.yml" -o -name "*.sql" -o -name "*.md" \
  | while read f; do
    if head -c 3 "$f" | xxd | head -1 | grep -q "efbb bf"; then
      echo "❌ BOM 头: $f"
      exit 1
    fi
    if ! file -i "$f" | grep -qE "charset=(utf-8|us-ascii)"; then
      echo "❌ 非 UTF-8: $f ($(file -i $f | cut -d= -f2))"
      exit 1
    fi
  done

# 2. application.yml 必须含 server.servlet.encoding.force=true
grep -q "force: true" plm-admin/src/main/resources/application.yml \
  || { echo "❌ application.yml 缺 force: true"; exit 1; }

# 3. Maven pom 必须 sourceEncoding=UTF-8
grep -q "<project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>" pom.xml \
  || { echo "❌ pom.xml 缺 sourceEncoding UTF-8"; exit 1; }

echo "✅ 字符编码配置全过"
```

## 4. 运行期自检（E2E 必跑）

每次部署后跑：

```bash
TOKEN=$(login_and_get_token)
# 写入含中文的记录
curl -s -X POST http://localhost:8081/business/requirement \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json; charset=UTF-8" \
  -d '{"projectId":1,"title":"编码自检测试 - 你好 αβγ"}'

# 读 DB HEX 校验
HEX=$(mysql -uroot -pXXX --default-character-set=utf8mb4 plm -Nse \
  "SELECT HEX(title) FROM tb_requirement WHERE title LIKE '%编码自检%' ORDER BY requirement_id DESC LIMIT 1")

# 预期开头是 E7BC96 E7A081 (编码) — 不是 EFBFBD
if echo "$HEX" | grep -q "^E7BC96E7A081"; then
  echo "✅ 编码自检通过"
else
  echo "❌ 乱码! HEX=$HEX"
  exit 1
fi
```

## 5. 已知坑速查

| # | 症状 | 根因 | 修复 |
|---|---|---|---|
| 1 | DB HEX 出现 `EFBFBD` (4 字节 U+FFFD 替换符) | HTTP 请求体被 GBK 解码 → 失败字节变 � → 再 UTF-8 编码存 DB | `-Dfile.encoding=UTF-8` + `server.servlet.encoding.force: true` |
| 2 | DB HEX 出现 GBK 字节（如 `B5BC` for "导"）但表是 utf8mb4 | JDBC URL 缺 `characterEncoding=utf8` | 修 URL |
| 3 | mysql CLI 显示乱码但 DB HEX 正确 | 控制台 code page 是 GBK | mysql 加 `--default-character-set=utf8mb4`，PowerShell `chcp 65001` |
| 4 | curl 发请求乱码（Windows MSYS）| MSYS 转换 args，bash heredoc UTF-8 → cmd 调 curl 时编码改变 | 用文件写 body：`curl -d @body.json` |
| 5 | 前端显示乱码但 API 返回正确 | 浏览器猜错字符集 | 后端 `Content-Type: application/json; charset=UTF-8` 显式 |
| 6 | SQL 文件导入后变乱码 | mysql CLI 默认 GBK 转换 | `mysql --default-character-set=utf8mb4 < file.sql` |
| 7 | java 控制台输出中文是 ??? | System.out 用 file.encoding | `-Dstdout.encoding=UTF-8 -Dstderr.encoding=UTF-8` |
| 8 | Excel 导出乱码 | POI 默认 ISO-8859-1 在 header Content-Disposition | URLEncoder.encode(filename, "UTF-8") |

## 6. 故障恢复流程

发现某记录已乱码：

1. **不可逆，必须删除**：`EFBFBD` 是原字节丢失的替换符，**无法恢复原始内容**
2. 用 `DELETE WHERE HEX(field) LIKE '%EFBFBD%'` 清掉脏数据
3. 修复编码配置后重新插入正确数据
4. **乱码痕迹永久留存的话**：DB 备份还原前的版本 + 重放正确数据
