## 0. 命名总纲

| 对象 | 风格 | 示例 |
|---|---|---|
| Java 包 | `cn.com.bosssfot.dv.plm.*` 起根；业务包必须在 `.system.business.<entity>` 下（除非显式新建 `plm-business` 模块）| `cn.com.bosssfot.dv.plm.system.business.project.domain` |
| Java 类 | UpperCamelCase；Domain 与表同名首字母大写；Mapper/Service/Controller 后缀固定 | `Project / ProjectMapper / IProjectService / ProjectServiceImpl / ProjectController` |
| Java 方法 | lowerCamelCase；查询用 `select…/get…/list…`；写入 `insert…/update…/delete…/save…` | `selectProjectList / insertProject` |
| Java 字段 | lowerCamelCase；数据库列 → Java 字段走标准驼峰转换 | `manager_user_id` ↔ `managerUserId` |
| DB 表 | 业务表 `tb_<entity_snake>`；系统表沿用 `sys_*`；Quartz 表保持 `qrtz_*` | `tb_project / tb_change_request / sys_user` |
| DB 列 | snake_case；外键 `<table>_id`；逻辑删除 `del_flag`（0=正常 / 2=删除） | `manager_user_id / del_flag` |
| TS/Vue 文件 | 组件文件夹 kebab-case，组件文件 `index.vue`；非组件 ts/tsx 用 kebab-case | `views/business/project/index.vue / api/business/project.ts` |
| Vue 组件名 | PascalCase（在 `<script setup name="..." >` 显式声明，便于 keep-alive） | `name="Project"` |
| 路由路径 | kebab-case，业务路由统一 `/business/<entity>` | `/business/project` |
| 权限串 | `<domain>:<entity>:<action>`，业务统一 `business:*`；系统沿用 `system:*` / `monitor:*` / `tool:*` | `business:project:list / business:project:export` |
| 字典 type | 业务字典强制 `biz_<entity>_<field>` 前缀，避免与 `sys_*` 冲突 | `biz_project_type / biz_project_status` |
| 环境变量 | UPPER_SNAKE_CASE，按服务分组前缀 | `JWT_SECRET / DB_PASSWORD / REDIS_HOST` |

## 1. 后端规范（Spring Boot / MyBatis）

### 1.1 目录与分层

业务模块**默认放在** `plm-system` 子包 `business/`：

```
plm-system/src/main/java/<package-path>/system/business/<entity>/
  domain/<Entity>.java
  mapper/<Entity>Mapper.java
  service/I<Entity>Service.java
  service/impl/<Entity>ServiceImpl.java
plm-system/src/main/resources/mapper/business/<Entity>Mapper.xml

plm-admin/src/main/java/<package-path>/web/controller/business/<Entity>Controller.java
```

**Controller 必须在 `plm-admin/web/controller/`** 下（保持 RuoYi 惯例：admin 模块承载 web 层）。

> 由 [ruoyi-bootstrap](https://) skill 的 Phase 7 一键生成符合本结构的骨架。

### 1.2 Controller 模板（标杆示例）

```java
@RestController
@RequestMapping("/business/project")
public class ProjectController extends BaseController {
    @Autowired private IProjectService projectService;

    @PreAuthorize("@ss.hasPermi('business:project:list')")
    @GetMapping("/list")
    public TableDataInfo list(Project project) {
        startPage();                                       // 分页由 PageHelper 处理
        return getDataTable(projectService.selectProjectList(project));
    }

    @PreAuthorize("@ss.hasPermi('business:project:export')")
    @Log(title = "项目", businessType = BusinessType.EXPORT)
    @PostMapping("/export")
    public void export(HttpServletResponse response, Project project) {
        List<Project> list = projectService.selectProjectList(project);
        new ExcelUtil<>(Project.class).exportExcel(response, list, "项目数据");
    }

    @PreAuthorize("@ss.hasPermi('business:project:query')")
    @GetMapping("/{id}")
    public AjaxResult getInfo(@PathVariable Long id) {
        return success(projectService.selectProjectById(id));
    }

    @PreAuthorize("@ss.hasPermi('business:project:add')")
    @Log(title = "项目", businessType = BusinessType.INSERT)
    @PostMapping
    public AjaxResult add(@RequestBody Project project) {
        return toAjax(projectService.insertProject(project));
    }

    @PreAuthorize("@ss.hasPermi('business:project:edit')")
    @Log(title = "项目", businessType = BusinessType.UPDATE)
    @PutMapping
    public AjaxResult edit(@RequestBody Project project) {
        return toAjax(projectService.updateProject(project));
    }

    @PreAuthorize("@ss.hasPermi('business:project:remove')")
    @Log(title = "项目", businessType = BusinessType.DELETE)
    @DeleteMapping("/{ids}")
    public AjaxResult remove(@PathVariable Long[] ids) {
        return toAjax(projectService.deleteProjectByIds(ids));
    }
}
```

Controller 规则（必须遵守）：

- **每个写操作必须有 `@Log` 注解**（INSERT / UPDATE / DELETE / EXPORT / IMPORT / GRANT），写入操作日志便于审计。
- **每个端点必须有 `@PreAuthorize` 权限注解**，永远不要"忘了加权限"。
- **Controller 只做参数装配 + 调用 Service + 包装返回**，不写业务逻辑（业务在 Service 里）。
- **不要在 Controller 里 `try/catch` 业务异常**；让 `GlobalExceptionHandler` 统一处理。
- **不要返回原始 `List<T>`**；统一用 `AjaxResult` / `TableDataInfo` 包装。

### 1.3 Service 规则

- **Service 接口 + 实现成对存在**：`IProjectService` + `ProjectServiceImpl`。即便没多实现也保留接口，方便后续 Mock / 替换。
- **写入操作必须设置审计字段**：
  ```java
  project.setCreateBy(SecurityUtils.getUsername());   // insert 时
  project.setUpdateBy(SecurityUtils.getUsername());   // update 时
  ```
  `create_time / update_time` 走 MyBatis 的 `sysdate()`（已在模板 XML 中设好）。
- **跨表写入必须 `@Transactional(rollbackFor = Exception.class)`**：默认只回滚 RuntimeException，业务异常常被吞掉，必须显式声明。
- **读操作不要加事务**。
- **Service 内不要直接调 HttpServletRequest / Response**；那是 Controller 的事。

### 1.4 Mapper 规则

- **Mapper 接口** 只声明方法，不带 SQL 注解（保持 SQL 在 XML 里）。
- **XML 列 / 属性映射** 用 `<resultMap>`，不要依赖 MyBatis 自动驼峰转换（容易踩坑）。
- **List 查询必须有 `<where>`**，每个条件 `<if test="X != null and X != ''">`，避免 `and` 漏掉。
- **逻辑删除统一通过 `del_flag = '2'`**，永远不要 `DELETE FROM`：
  ```xml
  <update id="deleteProjectByIds" parameterType="Long">
      update tb_project set del_flag = '2' where id in
      <foreach item="id" collection="array" open="(" separator="," close=")">#{id}</foreach>
  </update>
  ```
- **INSERT 必须 `<trim>` 包字段列表**，避免 null 字段写空字符串污染数据。
- **绝不拼接用户输入到 SQL 字符串**；任何动态 SQL 走 `#{param}`，需要表名/列名动态时也必须白名单校验。

### 1.5 Domain 规则

- **业务实体继承 `BaseEntity`**，自动得到 `createBy / createTime / updateBy / updateTime / remark / params`。
- **导出字段加 `@Excel(name = "...")`**；关联字典字段加 `dictType = "biz_xxx_yyy"`，导出时自动翻译。
- **日期字段加 `@JsonFormat(pattern = "yyyy-MM-dd")`**（具体精度按业务）。
- **金额类用 `BigDecimal`**，不要 `double / float`。
- **`toString()` 用 `ToStringBuilder` 风格**（与若依模板一致，便于日志查看）。

### 1.6 异常与错误码

- **抛业务异常用 `ServiceException(msg)` 或 `ServiceException(msg, code)`**；
- **错误码段位**（约定）：
  | 段位 | 用途 |
  |---|---|
  | 200 | 成功 |
  | 401 | 未登录 |
  | 403 | 无权限 |
  | 404 | 资源不存在 |
  | 500 | 系统异常 |
  | 600 - 699 | 业务参数错误 |
  | 700 - 799 | 业务规则错误（如状态不允许操作） |
  | 800 - 899 | 外部依赖错误（第三方 API / DB） |

  错误码常量集中放在 `<package>.common.constant.ErrorCode`（如不存在则新建）。

- **绝不在业务代码里 `e.printStackTrace()`**；用 `log.error("msg", e)`，让 logback 接管。
- **绝不 `catch (Exception e) { }`**（吞异常）；至少 log + 抛出或包装成 ServiceException。

### 1.7 日志规范

- **使用 `org.slf4j.Logger`**（项目通过 `@Slf4j` 或手动 `LoggerFactory.getLogger(getClass())`）；不要用 `System.out.println`。
- **日志级别**：
  - `ERROR`：系统故障 / 业务流程中断
  - `WARN`：可恢复异常 / 配置降级
  - `INFO`：业务关键节点（登录、下单、关键状态变更）
  - `DEBUG`：开发期排查；生产关闭
- **日志参数化**：`log.info("project {} status changed to {}", id, status);`，**不要字符串拼接**。
- **敏感字段必须脱敏**：手机号、身份证、密码字段在日志中必须打码。

### 1.8 事务边界示例

```java
@Override
@Transactional(rollbackFor = Exception.class)
public int approveChangeRequest(Long crId, String approver) {
    ChangeRequest cr = mapper.selectById(crId);
    if (cr == null) throw new ServiceException("变更不存在", 604);
    if (!"pending".equals(cr.getStatus())) throw new ServiceException("当前状态不允许审批", 701);

    cr.setStatus("approved");
    cr.setApprover(approver);
    int rows = mapper.update(cr);

    historyMapper.insert(History.of(cr, "approved"));   // 跨表写入
    return rows;
}
```

## 2. 前端规范（Vue 3 + Vite + TS）

> **视觉/交互规范**：完整 UED 规范（颜色 Token、组件样式、AI 按钮规则、无障碍）见 [02-设计/UED规范.md](../02-设计/UED规范.md)。Claude 执行约束见 [.claude/rules.md §N](../.claude/rules.md)。本节只补充代码组织和工程约定。

### 2.1 目录组织

```
plm-frontend/src/
  api/business/<entity>.ts                  # 接口调用
  types/api/business/<entity>.ts            # 接口类型
  views/business/<entity>/index.vue         # 列表 + CRUD 主页面
  views/business/<entity>/components/*.vue  # 该模块专属子组件
  components/                               # 跨业务通用组件
  utils/                                    # 跨业务工具函数
```

### 2.2 Vue SFC 风格

- **强制使用 `<script setup lang="ts">`**，不再写 Options API。
- **组件 name 显式声明** 便于 KeepAlive：`<script setup name="Project" lang="ts">`。
- **三段顺序统一**：`<template>` → `<script setup>` → `<style scoped>`。
- **样式默认 `scoped`**，避免全局污染；只有全局 token / 重置样式才放 `src/assets/styles/`。
- **不要在 template 里写复杂表达式**，超过一层三元就提到 computed。

### 2.3 API 调用规范

- **所有调用走 `@/utils/request`**（已内置 token 注入、统一错误处理）。
- **每个业务模块独立的 `api/business/<entity>.ts`**，导出函数命名 `list<Entity> / get<Entity> / add<Entity> / update<Entity> / del<Entity>`。
- **入参类型用 `types/api/business/<entity>.ts` 的 `Form / Query`**，不裸 `any`。
- **永远不要在 Vue 组件里直接调 axios**。

### 2.4 类型规范

- **`types/api/common.ts` 已含 `BaseEntity` 与 `PageQuery`**，业务接口继承它们。
- **API 返回不要写 `any`**，至少 `{ code: number; msg: string; data: T; }` 或项目已有的 `ApiResponse<T>`。
- **慎用 `as unknown as T`**，需要时附短注释说明为什么。

### 2.5 权限控制

- **按钮级用 `v-hasPermi`**：`<el-button v-hasPermi="['business:project:edit']">修改</el-button>`。
- **路由级**：菜单 SQL 配置好后会自动从 sys_menu 加载，不需要前端硬编码。
- **角色级**：`v-hasRole="['admin']"`，谨慎使用，优先用 perm。

### 2.6 字典使用

- **不要在前端写死下拉选项**，用 `useDict('biz_xxx_yyy')`。
- **展示用 `<dict-tag :options="..." :value="..." />`**，带颜色标签。
- 字典 type 必须在 SQL 里 INSERT 到 `sys_dict_type` + `sys_dict_data`（业务模板 SQL 已含）。

## 3. SQL 规范

### 3.1 文件命名（在 `plm-backend/sql/`）

| 类型 | 命名 |
|---|---|
| 若依官方种子（请勿改）| `ry_<日期>.sql` |
| Quartz 系统表（请勿改）| `quartz.sql` |
| 业务表初始化 | `business-<entity>.sql`（每个实体一份） |
| 数据迁移脚本（如引入 Flyway / Liquibase）| `V<NN>__<描述>.sql` |
| 临时修复脚本 | `hotfix-<日期>-<描述>.sql` |

### 3.2 内容约束

- **必须 utf8mb4 + utf8mb4_0900_ai_ci**。
- **CREATE TABLE 列必须有 `comment`**。
- **变更列前必须备份**：
  ```sql
  -- 备份
  CREATE TABLE tb_project_bak_20260520 AS SELECT * FROM tb_project;
  -- 变更
  ALTER TABLE tb_project ADD COLUMN priority TINYINT DEFAULT 0 COMMENT '优先级';
  ```
- **删表 / 截断 / 删字段** 全部要走 PR 评审，commit message 用 `chore(db): drop column …` 明确。

### 3.3 导入约定

参见根 [CLAUDE.md](../CLAUDE.md) "Running locally"：**任何 SQL 导入必须 `mysql --default-character-set=utf8mb4 ...`**（gotcha #2）。

## 4. 提交与分支

### 4.1 Conventional Commits（强制，由 `.githooks/commit-msg` 校验）

```
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

`type`：`feat / fix / docs / refactor / test / chore / perf / build / ci / revert / style`
`scope`：业务模块名（`project / change-request / auth ...`）或基础设施名（`db / ci / docs / deps`）
`subject`：动词起头，≤ 70 字符，结尾不要句号

**Breaking change** 在 footer 写 `BREAKING CHANGE: <说明>`，会触发版本 major 升。

示例（好）：
- `feat(project): support batch close for projects`
- `fix(auth): JWT expiry timezone issue causing midnight relogin`
- `chore(deps): bump fastjson2 from 2.0.60 to 2.0.61`

反例（不好）：
- `update code` → 不知道你改了啥
- `修复bug` → 没有 type，没有 scope
- `feat: 重构整个登录` → 用 refactor 不是 feat；scope 缺失

### 4.2 分支模型

- `main`：保护分支，仅 PR 合入；CI 必须绿。
- `feature/<id>-<short-desc>`：新功能；`<id>` 是 issue / 任务编号。
- `fix/<id>-<short-desc>`：bug 修复。
- `hotfix/<short-desc>`：线上紧急修复；直接合 main 并打 tag。
- `release/x.y`：发版冻结分支，只接 cherry-pick 的 fix。
- 长期废弃分支每月清理。

### 4.3 PR 规范

每个 PR 必须包含：

- **背景**：解决什么问题（链接 issue 或 PRD 章节）
- **改动点**：列要点，不要罗列每个文件
- **自测情况**：本地跑过的命令 / 截图
- **关联**：相关 PR / ADR / 测试用例
- **风险**：可能影响的其他模块、回滚方式

合并要求：

- 至少 1 个 approve（架构级改动至少 2 个）
- CI 全绿
- 不绕过权限（不要用 `--no-verify` 跳 hook）
- 不强推 main / release 分支

## 5. 测试规范

### 5.1 后端

- **Service 单元测试覆盖核心逻辑**：用 JUnit 5 + Mockito；测试类与被测类同包；类名 `<Class>Test`。
- **Controller 集成测试**：用 `@SpringBootTest` + `MockMvc`，验证 HTTP 层、权限、参数校验。
- **测试数据**：用 `@Sql` 加载 fixture SQL，不要直接连开发库。
- **覆盖率门槛**：核心 Service ≥ 70%；Controller ≥ 50%（由 jacoco 报告）。

### 5.2 前端

- **utils / composables 必须有单元测试**（Vitest）。
- **关键业务流走 Playwright E2E**：脚本放 `04-测试/automation/`（按需建子目录）。
- **快照测试谨慎用**（容易过拟合到样式细节）。

### 5.3 测试入口文档

详细测试计划 / 用例库见 [04-测试](../04-测试/)。

## 6. 文档规范

### 6.0 无文档不执行（MUST,2026-05-17 生效）

业务模块的任何**实质性修改**(加字段、改状态机、加 API、改业务规则)**必须先有对应 Phase 文档**。文档缺则**拒绝代码变更**,先补文档再写代码。

**每模块必备 5 份核心文档**(参考 Project 范本):

| Phase | 必产出 | 触发"必须补"的代码改动 |
|---|---|---|
| 01 立项 | `01-立项/<Module>-PRD.md` | 任何加字段/改业务规则 |
| 02 设计 | `02-设计/<Module>-数据库设计.md` | 改 SQL DDL / Domain / Mapper |
| 02 设计 | `02-设计/<Module>-API设计.md` | 改 Controller / 加端点 |
| 04 测试 | `04-测试/<Module>-测试计划-YYYY-MM-DD.md` + 测试用例库 | 加单测 / E2E |
| 05 上线 | `05-上线/<Module>-发布计划-YYYY-MM-DD.md` | 准备发布 / 改 Changelog |

例外:typo/lint/dead code 删除 + 紧急安全修复(7 天内补文档)。
完整流程见 [.claude/rules.md §O](../.claude/rules.md)。
模块文档完整性现状见 [99-跨阶段/audits/2026-05-17-process-docs-completeness-audit.md](../99-跨阶段/audits/2026-05-17-process-docs-completeness-audit.md)。

### 6.1 项目文档目录

所有项目文档放在 `0?-阶段名/` 或 `99-跨阶段/`（见根目录文件树）。**不要散落在源码目录里**。

### 6.2 文档命名

- 阶段产出物：在对应阶段目录下，文件名简洁中文（如 `PRD.md / 系统架构.md`）。
- 时序文档（周报、Sprint、会议）：`YYYY-MM-DD_<主题>.md` 或 `YYYY-WW-...`。
- 链接索引类：目录名带括号注明，里面放 README。

### 6.3 代码内文档

- **Java**：所有 public 方法必须 Javadoc（至少一行说明）；Domain 类用注释标 `@author / 业务字段含义`。
- **TS/Vue**：复杂 utils 必须 TSDoc；Vue 组件 props 必须有类型注释。
- **绝不**让自动生成的 ruoyi 注释 `@author ruoyi` 被批量替换（框架归属保留，见 [.claude/rules.md](../.claude/rules.md)）。

## 7. 安全红线

不可妥协的 5 条：

1. **永远不要 commit 真实密码 / token / API key 到 git**。本地 `.env` 已 gitignore，CI 走环境变量。
2. **所有写接口必须有权限注解**；GET 接口最低 `@PreAuthorize` 也要 `hasPermi('xxx:query')`。
3. **接收用户输入必须校验**：Bean Validation (`@NotBlank @Min @Pattern`) 或自定义 Validator。
4. **导出/下载接口必须限制范围**：不能让前端传 `path` 参数让后端读任意文件。
5. **依赖升级前查 CVE**：`mvn dependency:tree` + `npm audit`，重大漏洞优先级 P0。

每次发布前过一遍 [04-测试/安全审计.md](../04-测试/安全审计.md) Checklist。
