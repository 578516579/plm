## 1. 测试金字塔(共 5 层)

```
                ┌───────────────────────┐
                │   性能/压测 (k6)        │  ← plm-perf-test/  (P95/P99 + 拐点)
                ├───────────────────────┤
                │   E2E (Playwright)     │  ← plm-frontend/e2e/  (浏览器真渲染)
                ├───────────────────────┤
                │   API 契约 (MockMvc)   │  ← <module>/src/test/.../controller/  (HTTP↔Java)
                ├───────────────────────┤
                │   组件 (Vitest)        │  ← plm-frontend/.../__tests__/  (字典 + 业务组件)
                ├───────────────────────┤
                │   单元 (JUnit5+Mockito)│  ← <module>/src/test/.../service/impl/  (业务逻辑)
                └───────────────────────┘
              数量↑    耗时↑     依赖外部↑
```

| 层 | 框架 | 启动依赖 | 跑一次耗时 | 覆盖目标 |
|---|---|---|---|---|
| **单元** | JUnit 5 + Mockito | 无 | < 5s/模块 | 业务规则、状态机、字段校验、ADR(如 0001 编号) |
| **组件** | Vitest + Vue Test Utils + MSW | 无 | < 10s/模块 | 字典 label/tag、AiButton 等可独立测的 UI 单元 |
| **API 契约** | SpringBootTest + standalone MockMvc | 无 | < 15s/模块 | HTTP 方法/路径/参数绑定/JSON 序列化/AjaxResult code |
| **E2E** | Playwright(workers=1)| 后端 8081 + 前端 :80 + MySQL + Redis | ~30s/spec | 关键流(登录、列表、新增、状态变更),编码守门员 |
| **性能/压测** | k6 | 同 E2E 全栈 | 5min-1h | P95/P99、错误率、撞号率、状态机并发冲突 |

## 2. 各层框架选型决策记录

### 2.1 API 契约层 — 为什么用 standalone MockMvc 而不是 @SpringBootTest

| 选项 | 启动耗时 | 真实度 | 适用场景 |
|---|---|---|---|
| **standalone MockMvc** ✅ | < 2s/模块 | 中等(无 Security/AOP)| 默认选择,聚焦 HTTP↔Java 契约 |
| @WebMvcTest 切片 | 5-10s | 高(含部分 auto-config)| 当需要测 @PreAuthorize 时 |
| @SpringBootTest | 30s+(本项目 38 模块)| 最高 | 仅集成测试/Phase 06 staging 用 |

决策来源:[plm-project/src/test/.../ProjectServiceImplLightIntegrationTest 头部 javadoc](../plm-backend/plm-project/src/test/java/cn/com/bosssfot/dv/plm/project/service/impl/ProjectServiceImplLightIntegrationTest.java) 已为 Service 层选了"Mockito 不启 Spring";本规范延续到 Controller 层。

需要额外补的(测试时手动注入):
- `org.springframework:spring-webmvc` test scope(标准业务模块 pom 默认不带 web)
- inline `@RestControllerAdvice` 模拟 GlobalExceptionHandler 把 ServiceException 翻译为 AjaxResult(plm-framework 在业务模块测试 classpath 之外)

### 2.2 性能/压测层 — 为什么用 k6 而不是 JMeter / Gatling / Locust

| 候选 | 取舍 |
|---|---|
| **k6** ✅ | JS 脚本,与前端栈一致;CLI + Grafana 出品;指标可推 Prometheus;无 GUI 包袱 |
| JMeter | XML/.jmx 编辑麻烦,不适合 GitOps |
| Locust | Python 栈,引入额外维护成本 |
| Gatling | JVM 系最强但需 Scala/Kotlin DSL,学习曲线陡 |

决策来源:2026-05-28 测试范围对齐会(本规范)。

## 4. 触发命令清单

```powershell
# === 1. 后端单元测试 ===
cd plm-backend
mvn -pl plm-project test -Dtest=ProjectServiceImplTest

# === 2. 后端 API 契约测试 ===
mvn -pl plm-project test -Dtest=ProjectControllerContractTest

# === 3. 后端全模块覆盖率(opt-in profile)===
mvn -Pcoverage test
# 报告: <module>/target/site/jacoco/index.html

# === 4. 前端组件测试 ===
cd plm-frontend
npm run test:unit
# 单文件: npm run test:unit -- src/views/business/project/__tests__/projectDict.spec.ts

# === 5. E2E 全套 ===
# 前置:启动后端 8081 + 前端 :80 + MySQL + Redis(见 CLAUDE.md "Running locally")
cd plm-frontend
npx playwright test                       # 全部 headless
npx playwright test e2e/project.spec.ts   # 单 spec
npx playwright test --debug               # 单步调试
npx playwright test -g "状态机"           # 按用例名筛选

# === 6. 性能/压测 ===
# 前置 1:seed 测试数据(50 个 project)
cd plm-backend/sql/seed
$env:DB_PASSWORD='...'; .\seed-all.ps1 -Modules project

# 前置 2:启后端 + 前端 + MySQL + Redis
# 前置 3:k6 准备
cd plm-perf-test
Copy-Item env\local.env.example env\local.env  # 编辑账号密码

# 跑冒烟(1 VU × 30s)
k6 run --env-file env/local.env scenarios/smoke.js

# 跑负载(50 VU × 5min)
k6 run --env-file env/local.env scenarios/project-list-get.js

# 跑压力(100 VU × 2min,测撞号)
k6 run --env-file env/local.env scenarios/project-create-post.js

# 跑状态机并发(50 VU,共享 10 项目)
k6 run --env-file env/local.env scenarios/project-state-transition.js
```

## 5. Phase 04 Gate 强制项 — 模块"测试就绪"准入

| 强制项 | 验证方法 |
|---|---|
| **B.0 单测覆盖** | `mvn -pl <module> -Pcoverage test` 行覆盖 ≥ 60%(*ServiceImpl)|
| **B.1 API 契约** | 每个 Controller endpoint ≥ 1 case;**ServiceException 路径必测**(601/701 错误码穿透到 AjaxResult.code)|
| **B.2 E2E** | 列表 + 新增 + 状态变更 ≥ 1 case;含 1 个"编码守门员"断言(`encoding.spec.ts` 风格)|
| **B.3 性能基线** | 至少 1 个 k6 场景,P95 < 业务基线(见 plm-perf-test/README §性能基线);跑过的 HTML 报告归档至 Gate 实例附录 |
| **B.4 安全审计** | 权限注解 `@PreAuthorize` 路径全覆盖;SQL 注入测试由 Mybatis 参数化兜底,但 dynamic SQL 段(if/foreach)必须手测 |
| **B.5 测试数据** | seed-<module>.sql 存在,带 `remark='seed'` + `create_by='seed-runner'`,可被 seed-cleanup 反向清理 |

每个模块通过 Gate 后,在 `99-跨阶段/gate-checklists/instances/<module>/Phase04-测试-Gate-<YYYY-MM-DD>.md` 落盘审计。
