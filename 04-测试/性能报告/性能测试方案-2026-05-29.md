## 1. 测试目标(SLO)

PLM 是内部协作工具,DAU 量级 < 1000,以**业务计算密集**(状态机校验、FK 校验、多表 join 列表、AI 调用)为主而非高并发。SLO 按"健康内部系统"档:

| 类别 | 关键接口 | P50 | P95 | P99 | 错误率 | 备注 |
|---|---|---|---|---|---|---|
| **登录** | `POST /login` | < 200ms | < 500ms | < 1s | < 0.1% | 含 captcha + JWT 签发 |
| **认证后元数据** | `GET /getInfo` / `getRouters` | < 100ms | < 300ms | < 500ms | < 0.1% | Redis 命中 |
| **业务列表** | `GET /business/<entity>/list?pageSize=10` | < 200ms | < 600ms | < 1s | < 0.5% | MyBatis 分页 + 1-2 join |
| **业务详情** | `GET /business/<entity>/<id>` | < 100ms | < 300ms | < 500ms | < 0.1% | 单表 PK 查询 |
| **业务写入** | `POST /business/<entity>` | < 300ms | < 800ms | < 1.5s | < 0.5% | 状态机校验 + FK 校验 + 事务 |
| **复合视图** | `GET /dashboard/summary` / `task/board` | < 500ms | < 1.5s | < 3s | < 0.5% | 多表聚合 |
| **AI 调用** | `POST /requirement/ai/evaluate` | 不在 SLO 内 | — | — | — | 受 LLM 厂商延迟支配,只统计可用性 |

**容量目标**:50 并发用户(≈3× 预期 DAU 同时在线),5 分钟稳压无 OOM、无 GC pause > 500ms、DB 连接池 ≤ 8(application-druid.yml 默认 max-active)。

## 2. 工具选型

| 工具 | 选 | 理由 |
|---|---|---|
| **k6** | ✅ 推荐 | JS 脚本(团队前端 Vue 熟),CLI 友好,出 JSON/CSV 易接 CI;不需要 GUI |
| JMeter | ⚠️ 备选 | 图形化 GUI 学习曲线高,XML 脚本难版本控制;只在需要复杂混合场景时用 |
| wrk | ⚠️ 备选 | 极致性能但 Lua 脚本写复杂场景麻烦 |
| Locust | ⚠️ 备选 | Python 团队若有 Python 栈可选,本项目无 |
| Gatling | ❌ 不选 | Scala 学习成本高 |

**结论**:首期用 **k6 0.50+**。脚本放 [`04-测试/性能报告/k6/`](k6/)(待建)。

## 3. 测试场景(本期 3 组样本)

### 3.1 场景 P-LOGIN — 登录基线

| 项 | 值 |
|---|---|
| URL | `POST {{base}}/login` |
| 负载 | 50 VU(virtual user)× 5min,ramp-up 30s |
| 数据 | 50 个预制用户 `perfuser01..50`/`Perf@123`(seed 脚本生成,见 §6) |
| 关键断言 | HTTP 200 + body.code=200 + body.token 非空 |
| 指标 | P95 < 500ms,错误率 < 0.1% |

### 3.2 场景 P-LIST — 业务列表读基线

| 项 | 值 |
|---|---|
| URL | `GET {{base}}/business/project/list?pageSize=10&pageNum=1` 等 6 个模块各 1 个 |
| 负载 | 30 VU × 10min,稳定 |
| 数据 | 每模块预灌 1000 行(seed 脚本) |
| 关键断言 | HTTP 200 + body.rows.length ≤ 10 + body.total > 0 |
| 指标 | P95 < 600ms,错误率 < 0.5% |

### 3.3 场景 P-WRITE — 业务写入(状态机+FK 校验)

| 项 | 值 |
|---|---|
| URL | `POST {{base}}/business/task` |
| 负载 | 20 VU × 5min,每 VU 串行写 |
| 数据 | 每次 POST 带不同 `taskNo`(VU id + 时间戳后缀)避 UK 冲突 |
| 关键断言 | HTTP 200 + body.code=200 + body.data.taskId > 0 |
| 指标 | P95 < 800ms,错误率 < 0.5% |
| 收尾 | afterEach cleanup,SQL DELETE 带前缀 `perf-` 的所有任务 |

## 4. k6 脚本骨架(示例)

```javascript
// 04-测试/性能报告/k6/login.k6.js
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 50 },
    { duration: '5m',  target: 50 },
    { duration: '30s', target: 0  },
  ],
  thresholds: {
    http_req_duration: ['p(95)<500', 'p(99)<1000'],
    http_req_failed:   ['rate<0.001'],
  },
};

const BASE = __ENV.PLM_BASE || 'http://localhost:8081';

export default function () {
  const captcha = http.get(`${BASE}/captchaImage`);
  const uuid    = JSON.parse(captcha.body).uuid;
  const vuId    = String(__VU).padStart(2, '0');
  const res = http.post(`${BASE}/login`, JSON.stringify({
    username: `perfuser${vuId}`,
    password: 'Perf@123',
    code:     '0',           // 与 captcha 验证关闭模式配合
    uuid,
  }), { headers: { 'Content-Type': 'application/json' } });
  check(res, {
    'status 200':    (r) => r.status === 200,
    'code 200':      (r) => JSON.parse(r.body).code === 200,
    'token present': (r) => !!JSON.parse(r.body).token,
  });
  sleep(1);
}
```

跑测命令:
```bash
cd 04-测试/性能报告/k6
k6 run --out json=login-2026-05-29.json login.k6.js
```

## 5. 测试环境与前提

| 项 | 配置 |
|---|---|
| 后端 | 单节点 JDK 17,`-Xms512m -Xmx2g -Dfile.encoding=UTF-8`(4 标志) |

## 7. 瓶颈分析 checklist

跑完报告必看的 6 项:

1. **JVM**: GC log(`-Xlog:gc*`)— Full GC 次数、单次 pause > 500ms?
2. **DB 连接池**: Druid console `active/poolingCount` — active 长期顶到 8 = 池耗尽,需调
3. **慢 SQL**: MySQL slow log(`long_query_time=0.5`)— 哪条 SQL 在 P95
4. **Redis**: `slowlog get 10` — 有无 BLPOP / KEYS 等阻塞命令
5. **CPU/内存**: 后端进程 `top -p <pid>` — CPU 长期 > 80% = 计算瓶颈
6. **Thread dump**: P99 漂的时段抓 `jstack <pid>` — 看是不是 BLOCKED on monitor

## 8. 改进建议路径(预案)

| 现象 | 短期 | 长期 |
|---|---|---|
| 列表 P95 > 600ms | 加索引、改 covering index | MyBatis lazy load + 缓存层 |
| 写入 P95 > 800ms | DB 连接池 max-active 8 → 32 | 异步化(@Async)非关键路径 |
| Full GC 频繁 | -Xmx2g → 4g | 排查内存泄漏(MAT 分析 heap dump) |
| Redis 慢 | 排查大 key | 拆 cluster |
| AI 调用慢 | 加超时 | LLM 切换/缓存高频 prompt |
