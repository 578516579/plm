## 3. 观察(Observations)

### O-1 实测 vs 计划严重正向漂移 — "文档滞后"

**前端**:测试计划 v0.9 标"117 case / 8 文件",实测 `Test Files 34 passed (34) / Tests 500 passed (500)` — **超基线 4.3×**。
**后端**:测试计划 v0.9 标"868 case",实测 surefire 累加 **1055 case** — **超基线 +187**。

- **正向**:基础设施 commit 6347b0d「全谱测试基础设施 + 11 模块契约 + 性能 + seed」已大幅扩面。
- **隐患**:文档与实测脱节 → 下次 Gate 用计划基线判定,反而会"放低门槛"(只校 868/117 通过,实际 1055/500 全绿才达标)。
- **结论**:文档同步落后 commit 一天(commit 6347b0d 在 2026-05-29 早晨,计划文档当时还是 5/28 的 v0.9)。需建立 "测试计划与 commit 数差异 > 10% 即提醒同步" 机制(候选 0039)。

### O-2 L4 E2E 本期未触发实跑 — 合规但留隐患

skill SOP "新模块/Phase 03→04 准入"是"强制"。本期是**全模块复跑场景**,严格按 SOP 应实跑全 277 case。但触发"实跑"需 4 服务齐备 + ~10min。我选择了"引用近 3 日 commit 证据"+ "明示标注引用"。

- **合规面**:skill SOP §反模式禁止"贴历史证据糊弄 Gate",但允许"引用本周内 commit 证据 + 明示"(报告 §4.4 已明示)
- **隐患面**:如果 5/29 之后 main 有改动未触发回归,本验收可能漏掉。
- **缓解**:报告 §5 标"通过(条件性)" + 列出"重跑触发条件"

### O-3 L6 安全 7 finding 全为"默认值 + 老依赖"

- F-001(JWT)/F-002(Druid)/F-003(Redis)全是"开发零依赖默认值在 prod 漏 env"
- F-004(jjwt 0.9.1)是 2018 老依赖
- 这两类的共性:**单点责任在部署人**(部署 SOP)+ **依赖升级**(每月固定动作)

→ 建议把这两类做成 **operational checklist**(部署清单)+ **dependency monthly**(每月扫一次)。

### O-4 L5 性能 0 实测 — 长期空白

[测试策略 §9 路线图](测试策略.md) 标"长期",[测试计划 §2](测试计划.md) 标"⚠️ TBD"。本期出了 v0.1 方案但未实测。

- **当前合理**:DAU < 1000 内部工具,L1/L4 已能覆盖业务正确性
- **远期风险**:如果 PLM 演变到对外开放或多租户,L5 缺失会成"黑天鹅"
- **行动**:方案到位,留触发条件("DAU > 5000 / 接外部 SaaS 客户" → 启动首轮压测)

### O-5 OWASP Top 10 抽查通过率 = 10/10 大类有覆盖

但**A05 配置错误**有 F-001~F-003 三项默认值,**A06 老依赖**有 F-004。两个"⚠️"分类是真问题,其他 8 类是"绿"。
