## 1. 测试理念

### 1.1 为什么测

测试的核心价值不是「证明代码没问题」，而是**提前暴露问题并为未来变更兜底**。PLM 是农业全生命周期管理系统，业务规则密集（状态机、FK 约束、业务错误码 6xx/7xx、中文编码守门员），手工回归成本极高。自动化测试在这里扮演三重角色：

- **防线** — 任何一次改动触发 CI，encoding 守门员和全套 E2E 立即反馈，乱码/状态机被在分钟级别捕获。
- **质量** — Gate 机制强制每个模块在交测前跑通该模块的 spec；Phase 03→04 不允许带着失败测试进入测试阶段。
- **速度** — 完善的自动化让人工测试聚焦在探索性、边界性场景，而非回归基线，整体迭代节奏更快。

### 1.2 测什么

按优先级排序：

1. **PRD-aligned 业务硬规则** — 状态机合法/非法转换、业务错误码（601 非法状态转换、702 FK 不存在、703 单一活跃约束、704-708 等）、编号格式（PRJ/REQ/SPR/TASK/DEF/TC-YYYY-NNNN）、字典白名单。
2. **编码守门员** — 中文字符经 Vue→Axios→Spring Boot→MySQL 全链路后 DB HEX 不含 `EFBFBD`（UTF-8 替换字符，乱码的典型症状）。这是 P0 一票否决项。
3. **FK 引用完整性** — 跨模块外键：projectId、sprintId、requirementId 不存在时后端应返 702，前端应展示友好错误。
4. **核心功能路径** — CRUD 接口的幂等性、列表分页、搜索条件、菜单路由可达性。
5. **业务计算逻辑** — 纯函数：日期解析、数字格式化、树状结构转换、selectDictLabel 等工具函数。

### 1.3 测试金字塔的应用

PLM 遵循**倒三角成本原则**：底层测试便宜且快，应尽量多写；顶层 E2E 昂贵且慢，只覆盖关键路径。

具体分工：
- **单元测试（底层）** — 覆盖纯函数、服务层业务规则；不依赖外部进程；毫秒级。**写得越多越好**，这是最廉价的安全网。
- **组件/集成测试（中层）** — 验证组件挂载行为、Store 状态流转、Mock 后端的前端交互；秒级。
- **E2E 测试（顶层）** — 验证从浏览器 UI 到数据库的完整链路；分钟级。**只覆盖核心场景**，不要重复底层已覆盖的分支。

## 6. Gate 接入

### 6.1 规则来源

测试 Gate 强制规则来自 [.claude/rules.md §G.4](../.claude/rules.md)，属于**一票否决项（P0 级，不允许任何例外）**。Gate Checklist 模板：[99-跨阶段/gate-checklists/Phase04-测试-Gate.md](../99-跨阶段/gate-checklists/Phase04-测试-Gate.md)。

### 6.2 Phase 03 → Phase 04 准入硬约束

每个 PRD-aligned 模块必须满足以下全部条件方可进入测试阶段：

**必须全过（MUST — 不允许跳过）**

| 检查项 | 命令 / 证据 | 目标 |
|---|---|---|
| encoding 守门员 | `npm run test:e2e:encoding` | 6/6 全过，DB HEX 无 EFBFBD |
| 模块 E2E spec | `npx playwright test {module}.spec.ts` | 100% 全过 |
| 全套 E2E | `npm run test:e2e` | 277/277（无新增失败） |
| 后端 ServiceImpl 单测 | `mvn test -pl plm-{module}` | BUILD SUCCESS |

## 7. 编写测试的 5 条军规

### 军规 1 — Read 真实代码，不 mock 想象中的行为

在写测试前，**必须先阅读** `ServiceImpl.java` 或 `*.spec.ts` 被测单元的真实代码。Mock 要模拟的是实际 Mapper/API 的行为，而不是你认为它应该返回什么。错误的 Mock 导致测试永远绿色，却在集成时爆炸。

```java
// 正确：先看真实的 sprintMapper.selectSprintByProjectIdAndStatus
// 再 mock 它返回 1 条（模拟"已有活跃 sprint"的场景）
when(sprintMapper.selectSprintByProjectIdAndStatus(projectId, "01")).thenReturn(List.of(existing));
```

### 军规 2 — 业务硬规则必须有专门 case 覆盖

每条业务错误码（6xx/7xx）和状态机边界都必须有对应的 test case，不能依靠"顺手覆盖"：

```java
@Test
void test_createSprint_703_singleActive() {
    // 这是 703 的专用 case，名字明确指向规则编号
    when(sprintMapper.selectSprintByProjectIdAndStatus(...)).thenReturn(List.of(active));
    assertThatThrownBy(() -> sprintService.createSprint(dto))
        .hasMessageContaining("703");
}
```

**清单：必须有专门 case 的规则**
- 601: 非法状态转换（每个模块的状态机每条非法边）
- 702: FK 引用不存在（projectId / sprintId / requirementId）
- 703: 单一活跃 Sprint 约束
- 704: Sprint 删除前置（有关联需求/任务时禁止删除）
- 705: Defect resolution 必填（关闭时）
- 706: TestCase 自动化脚本必填（is_automated=Y 时）
- 707/708: Submission 质量门禁
- 编号格式（ADR-000x 规范的正则检查）

### 军规 3 — 中文编码守门员每个模块都要有

每个业务模块的 E2E spec 必须包含至少 1 个 encoding case，直接调 `assertNoMojibake()`：

```typescript
test('TC-{Module}-Enc — 中文字段无乱码', async () => {
    const id = await api.create({ name: '测试中文名称' });
    await db.assertNoMojibake('plm_{table}', 'name', id);
});
```

这样即使 `encoding.spec.ts` 的 6 个通用 case 都过了，模块特定字段（如 sprint.goal、task.description）的乱码也能被模块级 spec 拦住。

### 军规 4 — 不要硬编码数据库密码

所有直连 MySQL 的测试代码（`helpers/db.ts`、Spring Test `@SpringBootTest`）必须从环境变量读取密码：

```typescript
// plm-frontend/e2e/helpers/db.ts
const password = process.env.DB_PASSWORD;
if (!password) throw new Error('DB_PASSWORD not set');
```

```java
// Spring Boot 4 会自动读 ${DB_PASSWORD} — 不要覆盖为硬编码
@Value("${spring.datasource.password}")
private String dsPassword; // ← 来自 application-druid.yml → env var
```

违反此规则会触发 rules §C 的安全否决。

### 军规 5 — 测试名用 TC-{模块}-{编号}-{中文描述} 格式

```typescript
test('TC-Spr-F004 — 703 项目级单一活跃 sprint', async () => { … });
test('TC-Def-F007 — 反向边 03→01 需要 resolution', async () => { … });
```

```java
@Test
@DisplayName("TC-Task-F006: 6×6 状态机 — 非法边 01→03 应抛 601")
void test_updateTaskStatus_601_illegalTransition_from01to03() { … }
```

使用 `TC-{模块缩写}-{PRD 功能编号}` 格式，让 test report 输出可以直接追溯到 PRD §F 和 [PRD-MAPPING.md §2](../PRD-MAPPING.md)。

## 8. 已知问题 / 反模式

### 8.1 Vite 首次编译冷启动慢（E2E 最高风险）

**症状**: E2E 首个 spec 失败，报 `waiting for selector .xxx: Timeout 10000ms exceeded`，但不是 selector 漂移。

**原因**: Vite dev server 对 `packages/plm-*` 子包进行按需编译，首次冷启动 5-15s；默认 10s timeout 不足。

**修复**（已在 PR #3 落地）: `playwright.config.ts` 中全局 `actionTimeout: 60000` + 每个 `test()` 开头 `page.setDefaultTimeout(60000)`。

**规避**: E2E 跑之前先在浏览器手动访问一次被测页面，触发 Vite 预编译，消除冷启动。

### 8.2 Chromium 1223 偶发 Network service crashed

**症状**: E2E 中途失败，日志 `Error: page.goto: net::ERR_NETWORK_CHANGED` 或 `Network service crashed`。

**修复**:
```bash
# Windows 强制关闭所有 Chromium 进程
taskkill /F /IM chrome.exe /IM chromium.exe 2>/dev/null
# 重跑失败的 case
npx playwright test --last-failed
```

**规避**: 在 `playwright.config.ts` 加 `retries: 1`（CI 环境），本地调试不加 retries，优先修根因。

### 8.3 注入 cookie 后路由守卫尚未完成

**症状**: `auth.ts` 注入 JWT cookie 后立即跳转 `/project`，但页面仍在 `/login` 或白屏。

**原因**: Vue Router 的路由守卫在 `getRouters` API 返回后才注册动态路由，cookie 注入 → `router.push` 之间有异步 gap。

**修复（已在 helpers/auth.ts 中）**:
```typescript
await page.waitForURL(/\/\w/, { timeout: 15000 });
await page.waitForResponse(resp => resp.url().includes('/getRouters'));
await page.waitForTimeout(500); // DOM 渲染缓冲
```

### 8.4 DB_PASSWORD 未设置导致静默错误

**旧反模式**（已在 PR #1 修复）:
```typescript
password: process.env.DB_PASSWORD || 'admin123'  // ← 危险！静默忽略未设置
```

**正确模式**:
```typescript
const pwd = process.env.DB_PASSWORD;
if (!pwd) throw new Error('[db.ts] DB_PASSWORD env var not set. Export it first.');
```

### 8.5 多 worker 并发导致 DB 数据冲突

**症状**: 两个 spec 同时跑，互相看到对方的测试数据，断言失败（如列表数量不符）。

**修复（已在 `playwright.config.ts` 中）**: `workers: 1`（强制串行）。前端单元/组件测试不受此限制，可多 worker。

**隔离机制（fixtures.ts）**: 每个 spec 在 `beforeEach` 生成唯一 `RUN_ID = Date.now()`，作为所有测试数据字段的后缀，`afterEach` 清理带该 ID 的数据。

### 8.6 Windows JDK 默认编码 GBK

**症状**: 后端 `new String(bytes)` 或 JSON 序列化路径用了 Charset.defaultCharset()，写入 MySQL 的中文是 `?` 或 `EFBFBD` 乱码。

**修复**: 后端启动时加 4 个 JVM 标志（**不可省略任何一个**）：
```bash
java \
  -Dfile.encoding=UTF-8 \
  -Dsun.jnu.encoding=UTF-8 \
  -Dstdout.encoding=UTF-8 \
  -Dstderr.encoding=UTF-8 \
  -jar plm-admin.jar
```
