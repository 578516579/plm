## 5. 风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 后端冷启动慢，导致 captcha 拿不到 | 测试 timeout | [E2E-运行手册.md §1](测试用例库/E2E-运行手册.md) "等 60s 出 Started PlmApplication" |
| Redis 用 `localhost` (IPv6 坑) | Lettuce 超时 | 强制用 `127.0.0.1`（[CLAUDE.md 4 个坑](../CLAUDE.md)） |
| Windows JDK 默认 GBK | 乱码 EFBFBD | 后端启动 4 个 `-D` 标志（[字符编码规范.md](../03-开发/字符编码规范.md)） |
| 测试间共享 captcha 5min 过期 | 部分 case fail | worker=1 串行 + 每个 spec 独立 login（已在 [auth.ts](../plm-frontend/e2e/helpers/auth.ts)） |
| 多人并发跑测试 | DB 数据互相冲突 | RUN_ID 后缀 + 每 spec 独立 cleanup（[fixtures.ts](../plm-frontend/e2e/helpers/fixtures.ts)） |
| db.ts 误用硬编码密码 | 凭据泄露 | DB_PASSWORD 环境变量；无默认值，未设置直接抛错（已修复 2026-05-17） |

## 6. 执行流程（按文档跑测试 SOP）

按 [E2E-运行手册.md §1 三分钟跑通](测试用例库/E2E-运行手册.md) 执行：

```bash
# 步骤 0 — 金字塔下层(不依赖后端/前端进程,最快反馈)
cd plm-backend && mvn test --no-transfer-progress   # 期望 1055 case BUILD SUCCESS (~13min)
cd ../plm-frontend && npm run test:unit             # 期望 500 case / 34 文件 passed (~20s)

# 步骤 1 — E2E 前置自检（需后端 + 前端 + MySQL + Redis 已启动）
netstat -ano | grep -E ":3306|:6379|:8081|:80"   # 4 端口都 LISTENING
echo "$DB_PASSWORD" | head -c 5                   # 非空
curl -s http://localhost:8081/captchaImage | head -c 100   # JSON

# 步骤 2 — 跑 P0 守门员
cd plm-frontend
npm run test:e2e:encoding                         # 期望 6 passed (~20s)

# 步骤 3 — 跑 E2E 全套件
npm run test:e2e                                  # 期望 277 passed (~8-12min, workers=1 串行)

# 步骤 4 — 看结果
npm run test:e2e:report                           # HTML 报告
```

> 各层单跑命令速查（后端 `mvn test -pl plm-<module>`、前端 `npm run test:unit`、E2E 分组脚本）见 [测试策略.md §4](测试策略.md)。

如果失败，按 [E2E-运行手册.md §4](测试用例库/E2E-运行手册.md) 排查表诊断。
