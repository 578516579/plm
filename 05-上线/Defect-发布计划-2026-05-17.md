## 1. 发布范围
Defect 模块 v1.0 — 字段 + 状态机 + API + 字典 + 菜单全部就绪 (PRD-MAPPING §2 已对齐)。

## 2. 发布步骤
1. 数据库迁移: `mysql plm < sql/business-defect.sql`
2. 后端发布: `mvn -pl plm-defect install` + 重启 plm-admin
3. 前端发布: `npm run build` + 部署 plm-frontend
4. 烟雾测试: 通过菜单访问 + 创建 1 条数据 + 关键状态机演进
5. E2E 全套: `npm run test:e2e -g "Defect"`

## 3. 回滚预案
- DB: `mysql plm < sql/business-defect-rollback.sql`
- Code: git revert <feat commit>
- 菜单/字典:rollback SQL 自动清理

## 4. 监控

### 4.1 监控点 (3 维)

| 维度 | 指标 | 数据源 | 健康阈值 |
|---|---|---|---|
| **业务** | `tb_defect` 日新增行数 | MySQL count(create_time>=今日) | 中模块 50-500 / 日 |
| **业务** | P0 缺陷未关闭数 | MySQL count(priority='P0' AND status NOT IN ('closed','resolved')) | ≤ 3 |
| **业务** | 缺陷平均修复时长 (今日 closed) | MySQL avg(closed_time-create_time) where closed_time>=今日 | ≤ 48 小时 |
| **接口** | `/business/defect/*` 错误率 | Spring Actuator + Prometheus | 错误率 ≤ 1% |
| **接口** | `/business/defect/list` P95 响应时间 | APM | < 1000 ms |
| **资源** | tb_defect 表 size 增长率 | MySQL information_schema | 月增 < 15% |

### 4.2 告警阈值

| 等级 | 触发条件 | 通知渠道 | 处理 SLA |
|---|---|---|---|
| P0 | 接口 5xx 持续 > 5 分钟 | PagerDuty + 飞书 | 30 分钟响应 |
| P0 | P0 缺陷未关闭数 > 5 (积压告警) | PagerDuty + 飞书 | 30 分钟响应 |
| P1 | 列表查询 P95 > 3s 连续 10 分钟 | 飞书 | 2 小时响应 |
| P1 | 缺陷平均修复时长 > 96 小时 (滑动 7 天) | 飞书 | 2 小时响应 |
| P2 | 表行数月增 > 50% (异常增长) | 邮件 | 工作日响应 |

### 4.3 日志关键字

后端 Logback 输出, ELK / Loki 索引:
- `ERROR.*DefectService` — Service 层异常
- `WARN.*DefectController.*RejectedException` — 业务规则拒绝 (如未指派 fixer 尝试 closed)
- `INFO.*Defect.*state-transit` — 状态机推进审计 (new→confirmed→in-fix→resolved→verified→closed)
- `WARN.*Defect.*reopen` — 缺陷重开审计
- `WARN.*Defect.*deadlock` — MySQL 死锁
