## 1. 发布范围
Pipeline 模块 v1.0 — 字段 + 状态机 + API + 字典 + 菜单全部就绪 (PRD-MAPPING §2 已对齐)。

## 2. 发布步骤
1. 数据库迁移: `mysql plm < sql/business-pipeline.sql`
2. 后端发布: `mvn -pl plm-pipeline install` + 重启 plm-admin
3. 前端发布: `npm run build` + 部署 plm-frontend
4. 烟雾测试: 通过菜单访问 + 创建 1 条数据 + 关键状态机演进
5. E2E 全套: `npm run test:e2e -g "Pipeline"`

## 3. 回滚预案
- DB: `mysql plm < sql/business-pipeline-rollback.sql`
- Code: git revert <feat commit>
- 菜单/字典:rollback SQL 自动清理

## 4. 监控

### 4.1 监控点 (3 维)

| 维度 | 指标 | 数据源 | 健康阈值 |
|---|---|---|---|
| **业务** | `tb_pipeline` 日新增行数 | MySQL count(create_time>=今日) | 中模块 50-500 / 日 |
| **业务** | Pipeline 失败率 (今日) | MySQL count(status='failed') / count(last_run>=今日) | ≤ 10% |
| **业务** | Pipeline 执行卡顿 (queue 滞留 > 5 分钟) | MySQL count(status='queued' AND queue_time<NOW()-5m) | ≤ 5 |
| **接口** | `/business/pipeline/*` 错误率 | Spring Actuator + Prometheus | 错误率 ≤ 1% |
| **接口** | `/business/pipeline/list` P95 响应时间 | APM | < 1000 ms |
| **接口** | `/business/pipeline/trigger` P95 触发延迟 | APM | < 2000 ms |
| **资源** | tb_pipeline 表 size 增长率 | MySQL information_schema | 月增 < 20% |

### 4.2 告警阈值

| 等级 | 触发条件 | 通知渠道 | 处理 SLA |
|---|---|---|---|
| P0 | 接口 5xx 持续 > 5 分钟 | PagerDuty + 飞书 | 30 分钟响应 |
| P0 | Pipeline 失败率 > 30% (滑动 30 分钟, CI 系统故障) | PagerDuty + 飞书 | 30 分钟响应 |
| P1 | Pipeline 执行卡顿数 > 20 (队列积压) | 飞书 | 2 小时响应 |
| P1 | 列表查询 P95 > 3s 连续 10 分钟 | 飞书 | 2 小时响应 |
| P2 | 表行数月增 > 50% (异常增长) | 邮件 | 工作日响应 |

### 4.3 日志关键字

后端 Logback 输出, ELK / Loki 索引:
- `ERROR.*PipelineService` — Service 层异常
- `WARN.*PipelineController.*RejectedException` — 业务规则拒绝
- `INFO.*Pipeline.*state-transit` — 状态机推进审计 (queued→running→success/failed/cancelled)
- `ERROR.*Pipeline.*step-failed` — 步骤失败 (含 step / exit_code)
- `WARN.*Pipeline.*queue-slow` — 队列滞留告警
- `WARN.*Pipeline.*deadlock` — MySQL 死锁
