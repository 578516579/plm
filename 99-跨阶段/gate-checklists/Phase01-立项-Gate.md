## A. 准入条件（进入 Phase 01 前必须满足）

- [ ] 已有明确的业务痛点 / 机会描述（一句话能讲清）
- [ ] 已有初步价值假设（用户量、收入、成本节约等量级估计）
- [ ] 已识别该项目的最终决策人（拍板"是否做"的人）
- [ ] 列入团队 [OKR](../团队%20OKR.md) 候选清单 / 或已与 PM Owner 达成共识立项

---

## B. 必产出物（按顺序产出）

### B.1 市场调研（项目类型差异化）

- [ ] 文件：`01-立项/<模块>-市场调研.md`（基于 [模板](../../01-立项/市场调研.md)）
- [ ] 含：明确结论（值得做 / 边做边看 / 不值得做）— **所有类型必填**

#### 按 项目类型 差异化条目

**type=`external-product`**：
- [ ] 含：目标市场规模（TAM / SAM / SOM）
- [ ] 含：至少 3 个竞品的功能矩阵
- [ ] 含：用户访谈纪要 — 按团队规模阈值（见维度 3）

**type=`internal-tool`**：
- [ ] 含：现有工具 / 替代方案对比（至少 3 个，含飞书表格 / 自建 / SaaS 各类型）
- [ ] 含：内部用户痛点访谈 — 按团队规模阈值
- [ ] 含：决策"为何自建 vs 用已有"

**type=`framework-upgrade`**：
- [ ] 可省略本节（B.1 跳过）；用 [02-设计/ADR](../../03-开发/ADR/) 替代，强制写"现状痛点 / 升级动机 / 影响面"

### B.2 PRD（产品需求文档）

- [ ] 文件：`01-立项/<模块>-PRD.md`（基于 [模板](../../01-立项/PRD.md)）
- [ ] 含：背景与目标
- [ ] 含：核心衡量指标（北极星指标 / OKR KR 对应）
- [ ] 含：用户角色与典型场景
- [ ] 含：功能需求清单（带优先级 P0/P1/P2）
- [ ] 含：每条需求的验收标准
- [ ] 含：非功能需求（性能 / 安全 / 合规约束）
- [ ] 含：**Out of Scope 明确**（什么本期不做）
- [ ] 含：依赖与约束（上下游系统、第三方）
- [ ] PRD 版本号已分配（v1.0）

### B.3 商业计划（按 类型 + 级别 差异化）

- [ ] 文件：`01-立项/<模块>-商业计划.md`（仅 L1 强制；L2 可省略）

**type=`external-product`**：
- [ ] 含：价值主张
- [ ] 含：商业模式 + 收入预测
- [ ] 含：投入预算（人 / 基础设施 / 推广）
- [ ] 含：关键假设 + 验证方法（≥ 3 条）

**type=`internal-tool`**：
- [ ] 含：价值主张（解决谁的问题、为什么是我们）
- [ ] 含：**ROI 工时节约模型**（投入 vs 节约的对账表）+ 回本周期
- [ ] 含：关键假设 + 验证方法（≥ 3 条）
- [ ] 含：退出条件（什么情况下下线本模块）
- [ ] 不强求"收入预测"（内部工具不适用）

**type=`framework-upgrade`**：
- [ ] 可省略本节；用 ADR + Phase 02 影响面分析替代

### B.4 立项评审纪要（按 规模 差异化）

- [ ] 文件：`01-立项/<模块>-立项评审纪要-<YYYY-MM-DD>.md`
- [ ] 评审日期：YYYY-MM-DD
- [ ] 评审材料已提前发出（按团队规模）：
  - [ ] `solo` / `small`：0 天（当日 OK）
  - [ ] `medium`：≥ 1 工作日
  - [ ] `large`：≥ 2 工作日
- [ ] 评审结论明确（**通过 / 有条件通过 / 不通过**）

## C. Definition of Done（出口 DoD）

- [ ] PRD v1.0 已评审通过（结论"通过"或"有条件通过 + 条件已闭环"）
- [ ] 评审纪要已归档到 `01-立项/`
- [ ] 评审中识别的风险已 100% 录入 [99-跨阶段/风险登记册.md](../风险登记册.md)
- [ ] 已排上当前或下一季度 OKR（[99-跨阶段/团队 OKR.md](../团队%20OKR.md)）
- [ ] 已通知技术 lead（启动 Phase 02 设计的准入条件）
- [ ] **本 Checklist 文件已 commit 入库**（commit message：`docs(gate): <module> phase 01 passed`）

---

## D. 评审记录与签字（按 团队规模 调整必填角色数）

按团队规模需达成的最少签字角色数（见 [README §维度 3](../README.md)）：

- `solo`：1 个（自评，**commit message 必须带 `[solo-review]` 标签**，并在 E 段写明自评理由）
- `small`：2 个（含业务方 + 产品 OR 技术）
- `medium`：3 个（业务方 + 产品 + 技术）
- `large`：5 个（业务方 + 产品 + 技术 + 设计 + CFO/财务）

## L2 简化指引

L2 中型需求可按下列方式简化：
- **B.1 市场调研** → 可以是一段 200 字的"竞品速览 + 我们的差异化"，不强制 5 个访谈
- **B.3 商业计划** → 可省略；改为在 PRD 内加一段"投入产出预估"
- **D 评审签字** → 业务 + 产品 + 技术 lead 三方即可（不强求 CFO / 设计）

L3 小型需求**不走本 Gate**，直接走 Phase 03 PR 流程。
