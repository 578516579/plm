## 1. 背景

Project Phase 01 Gate 实例头部我标了"分级 = L1"，但没记"为什么是 L1"。3 个月后回看不知道是因为"新业务模块"还是"DB 结构性变更"还是"两个都满足"。

## 2. 证据

- 关联 reflect: [../reflect/2026-W20-project-phase01-dogfood.md](../reflect/2026-W20-project-phase01-dogfood.md) F10
- 实际例子: Phase 01 Gate 实例头部 "分级 = L1（PLM 第一个业务模块，按"高就高"判级；且引入新表 `tb_project` 属"DB 结构性变更"）" — 实际我写了理由在括号里，但模板没强制。容易漏。

## 3. 提案

在 6 个 PhaseNN-Gate.md 模板的头部"分级"字段下新增一行"分级理由（必填）"：

```diff
 | 分级 | L1 / L2 / L3 |
+| 分级理由 | _触发的具体条件，引用 README §分级 中的判定列_ |
```

实例填写示例：
```
| 分级 | L1 |
| 分级理由 | 新业务模块 + DB 结构性变更（新建 tb_project 表）→ 触发 L1 |
```

## 4. 改动文件

| 文件 | 改动 |
|---|---|
| `gate-checklists/Phase01-...Gate.md` ~ `Phase06-...Gate.md` 头部 | 加"分级理由"字段 |
| `gate-checklists/README.md` §使用流程 第 2 步"填头部信息" | 说明"分级理由"必填 |
| `gate-checklists/instances/README.md` 表格 | 列加"分级理由"（可省略，太长会撑爆表格）|

## 5. 影响

- 已存在的 Phase 01 实例：可补一行"分级理由"，走"修订记录"追加
- 新实例：填头部时必填

## 6. 风险

- 形式主义风险：填"因为 PRD 说 L1"这种废话。缓解：README §分级 给出"理由必须引用 README 中的判定列条目，如「新业务模块」「DB 结构性变更」"
