## 8. 测试编排（Test Orchestration）

> 由 `test-orchestrator` 每轮测试收口时记录（rules §G.5.4 MUST）。**flake 与真退步必须分开**。

| 字段 | 当月 | 说明 |
|---|---|---|
| e2e_flake_count | | 复测(`--retries=1`)后转绿的"假失败"次数 |
| e2e_real_fail_count | | 复测仍红的"真退步"次数 |
| coverage_gap | | 发现的覆盖缺口(哪层/哪模块) |
| rca_category | | 失败根因分布:env / schema / stale-jvm / code / contract / encoding |
| encoding_guard_breaks | | encoding 守门红的次数(应=0,P0) |
| gate_evidence_backfill_attempt | | 贴历史证据糊弄 Gate 的企图(应=0) |

> **触发提案条件**：同类 flake 月内 ≥3 → 测试稳定性提案；coverage_gap 反复同一层 → 补该层模板提案；encoding_guard_breaks > 0 → P0 复盘；gate_evidence_backfill_attempt > 0 → 严重违规调查。

---

## 9. 产品设计编排（Product Design Orchestration）

> 由 `product-orchestrator` 每轮产品设计编排收口时记录（rules §M.9.4 MUST）。衡量"开发前的设计是否对得上 PRD/原型"。

| 字段 | 当月 | 说明 |
|---|---|---|
| prd_drift_count | | 需求/代码与 PRD 不符被拦截的次数 |
| prototype_deviation_count | | 原型保真 §N(UED) 违规数 |
| untraceable_field_count | | 指不出 PRD/原型出处的字段/状态/错误码数（应趋 0） |
| field_table_lag | | 字段对照表晚于代码 commit 的次数（应=0,§M.2） |
| prd_change_via_proposal | | PRD 演化正确走 proposal 的占比（越高越合规） |

> **触发提案条件**：同类 prd_drift_count 月内 ≥3 → PRD-MAPPING 该模块补全/重审提案；untraceable_field 反复同类 → 原型补画/PRD 补章节提案；prototype_deviation 集中某 §N 子项 → 该 UED 约束加 hook/培训提案；field_table_lag > 0 → §M.2 复盘。

## 10. 数据库设计编排（Database Design Orchestration）

> 由 `db-orchestrator` 每轮数据库设计编排收口时记录（rules §M.10.4 MUST）。衡量"建表前的 schema 设计是否对得上 PRD 字段表 + 命名/charset 规约"。

| 字段 | 当月 | 说明 |
|---|---|---|
| schema_naming_drift_count | | 命名漂移被拦截次数（用了 creator_id 而非 author_user_id 等,§M.7） |
| index_gap_count | | 查询模式无索引覆盖被发现的次数 |
| charset_violation_count | | 非 utf8mb4 / 缺 `--default-character-set`（gotcha #2,应=0） |
| migration_unsafe_count | | 非幂等 / 大表裸 ALTER 被拦截次数 |
| missing_sys_menu_count | | business-*.sql 缺 sys_menu（gotcha #7,pre-commit hook 拦截数） |
| schema_drift_count | | DB 实际 schema 与 sql 期望不一致次数（db-ops 验证） |

> **触发提案条件**：`charset_violation_count` > 0 → **P0 复盘**（gotcha #2 复发）；`schema_naming_drift_count` 月内 ≥3 → §M.7 强化 / 加命名 lint hook 提案；`index_gap_count` 集中某类查询 → 补索引设计 checklist 提案；`missing_sys_menu_count` 反复 → 查 pre-commit hook 是否被 `--no-verify` 绕过。

## 11. UED 设计编排（UED Design Orchestration）

> 由 `ued-orchestrator` 每轮 UED 设计编排收口时记录（rules §N.10.4 MUST）。衡量"开发前的 UI/交互设计是否对得上 UED规范+原型+无障碍"。

| 字段 | 当月 | 说明 |
|---|---|---|
| ued_token_violation_count | | 裸十六进制 / 非 4px 倍数间距 / 匿名颜色违规数（应=0,§N.1/N.6） |
| component_reuse_gap | | 没用 UED规范库组件 / 用了未登记新 CSS 类的次数（应趋 0,§N.9） |
| a11y_violation_count | | WCAG AA 违规数：对比度 / focus-visible / label / 不靠颜色（§11） |
| three_state_miss_count | | 空 / 载 / 错三态缺失数（应=0,§N.5） |
| ued_handoff_lag | | 前端实现先于 UED 规格就绪的次数（应=0） |

> **触发提案条件**：同类 `ued_token_violation_count` 月内 ≥3 → 加 stylelint/PreToolUse hook 拦裸 hex/间距提案；`a11y_violation_count` 反复同一子项 → axe-core 纳入 E2E 提案；`component_reuse_gap` 反复 → UED规范补组件章节提案；`three_state_miss_count` 反复 → 组件模板默认带空/载/错骨架提案。
