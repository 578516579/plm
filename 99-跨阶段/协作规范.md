# 并行 Session 协作规范

> **适用范围**：仓库内任何并行作业场景 — 多个 Claude session（不同 worktree）、多个开发者、人机混合协作。
>
> **目标**：让 N 个 session 同时在仓库里干活，不撞 SSoT、不重复造轮子、合并顺畅。
>
> **关联文档**：
> - [模块工作流.md](模块工作流.md) — 单模块 6 阶段流转
> - [.claude/rules.md](../.claude/rules.md) — Claude 视角硬约束（§F 提交 / §G Gate / §L 自进化）
> - [03-开发/开发规范.md](../03-开发/开发规范.md) — 人类视角分支/PR 规则（§4.2 §4.3）
> - [proposals/0008-parallel-session-collaboration.md](proposals/0008-parallel-session-collaboration.md) — 本规范的设立提案
>
> **强度分级**：**MUST** = 必守；**SHOULD** = 默认遵守、有正当理由可破；**MAY** = 建议。

---

## 0. 背景 — 当前仓库已实际多 worktree 并行

可用 `git worktree list` 查看；典型时刻同时有 3-6 个 `claude/*` 分支在各自子目录工作（2026-05-17 时点为 5 个）。多 session 收益是并行加速，代价是：

| 场景 | 风险 |
|---|---|
| 两个 session 同时改 `PRD-MAPPING.md` / `.claude/rules.md` | merge 冲突 + SSoT 被某一边覆盖 |
| 两个 session 同时给同一空壳模块（如 `testdata`）落地 | 两份不一致的 SQL/Java 实现 |
| 同一阶段同一天给同一模块写 Gate 实例 | 文件名撞车、签字记录被覆盖 |
| 各自给 proposal 占编号 | 编号撞车 |
| 一个 session 还没合入字段对照表，另一个就基于"旧表"写代码 | SSoT 和代码 commit drift |

本规范定义"并行作业的契约"，把上面 5 类风险变成可预防而不是靠 merge 时痛苦修。

---

## 1. 隔离层 — Worktree 是默认作业单位（MUST）

- 不允许在 `main` 工作树上**直接编辑**业务/规范文件。`main` 工作树只用来 `git pull` 看 SSoT 最新版与回看历史。
- 每个并行任务 → 一个独立 worktree + 独立分支。Claude session 默认落 `.claude/worktrees/<adjective-name>/`，分支 `claude/<name>`。
- 人类的活分支命名仍走 [开发规范.md §4.2](../03-开发/开发规范.md) — `feature/<id>-<desc>` / `fix/<id>-<desc>` / `hotfix/<desc>` / `release/x.y`。
- 任何 session 启动后**第一件事**：
  ```bash
  git worktree list                         # 我在哪、还有谁在
  git fetch --all --prune                   # 同步远端
  git log --oneline origin/main..HEAD       # 我领先 main 多少
  git log --oneline HEAD..origin/main       # main 领先我多少
  ```
  若"main 在我之后又前进 ≥ 10 个 commit"或"我上次 commit > 24h" → **先 rebase**，再继续。

| [99-跨阶段/signals/&lt;当月&gt;.md](signals/) | 当月度量信号 |
| [99-跨阶段/AgriPLM-模块映射-2026-05-16.md](AgriPLM-模块映射-2026-05-16.md) | 模块路线图 |
| [99-跨阶段/PLM-路线图.md](PLM-路线图.md) | 整体路线图 |
| 根 [CLAUDE.md](../CLAUDE.md) | Claude 自动加载入口 |
| 任何 `gate-checklists/instances/*` 已签字实例 | 不可覆盖，只能追加"修订记录" |

**规则**：

- **触发 proposal**（按 [.claude/rules.md §L.2](../.claude/rules.md)）：修改 `PRD-MAPPING.md` / `.claude/rules.md` / `.claude/settings.json` / `模块工作流.md` / `开发规范.md` / gate-checklists 模板 → **必须**先开 proposal；直接 commit = 违规。
- **单 commit 原则**：触及任一 SSoT 文件的 commit 不要混入其他无关改动。便于事后 cherry-pick / revert / 重排。
- **串行 merge**：同一时刻只允许一个 session 把 SSoT 改动合入 main；后到者必须 `git fetch && git rebase origin/main` 重做。
- **撞行检测**：rebase 后若发现自己改的行同时被别人改过 → **停下来 AskUserQuestion**：选 (a) 沿用对方版本 / (b) 用我的版本 / (c) 合并双方意图。**禁止** Claude 自作主张取舍。

## 3. 任务认领台账（MUST）

每个并行 session 开工前**必须**先把自己接的活写进 [在途任务.md](在途任务.md)：

```
| 分支 | worktree | 模块/范围 | 状态 | Owner / Session | 触及 SSoT | 起始 | 备注 |
```

- 开工前 → 添加一行
- 中间状态变化（进行中 / 等评审 / 等合并 / 暂停） → 改"状态"列
- 完工合入 main → 挪到"已完成"段
- 放弃 → 挪到"已完成"或删行，备注写原因

**冲突预防**：你接的活和台账上某行"模块/范围"重叠 → 不开工，先 ping 该行 Owner 协调。

---

## 4. 同模块串行 — 业务模块改动并发限制（MUST）

任一业务模块（如 `requirement / testdata / mcp / project`）在任一时刻**只允许一个 session 改其代码 + SQL**：

- 文档类（PRD-MAPPING.md 该模块段、设计文档）允许与代码改动**并行**，但要按 §2 串行 merge。
- 测试 / E2E spec 允许与代码改动**并行**（不同文件）。
- 同模块的 Gate 实例（`gate-checklists/instances/<模块>/`）走专属 Owner，不允许两个 session 同时签字。

**怎么判断"有人在改"**：看 [在途任务.md](在途任务.md) "模块/范围"列；看 `git branch -r --contains <module>/` 是否有未合并分支。

例外：**P0 hotfix（L3）** 可以打断长任务，但合入前必须通知正在改该模块的 Owner，并在台账加一行 hotfix 记录。

## 5. 编号防撞 — Proposal / ADR / Gate 实例（MUST）

| 资源 | 编号规则 | 撞号怎么处理 |
|---|---|---|
| [proposals/NNNN-*.md](proposals/) | 4 位数字递增 | 起新 proposal 前 `ls 99-跨阶段/proposals/` 看最大号；撞了让位（后到的 +1，顶部备注"原计划编号 X"） |
| [03-开发/ADR/NNNN-*.md](../03-开发/ADR/) | 同上 | 同上 |
| [gate-checklists/instances/&lt;m&gt;/PhaseNN-阶段-Gate-YYYY-MM-DD.md](gate-checklists/instances/) | 日期 + 模块 + 阶段 | 同模块同阶段同一天有第二份 → 文件名加 `-v2` 后缀，**两份都保留**（一份初签、一份复签） |
| commit message scope `<entity>` | 模块名 | 同 scope 多 session 各自 commit → 按 §4 同模块串行（不允许） |

**操作上**：编号决策放在 commit 前最后一刻，不要在分支开头就占号（避免长任务占着号不放）。

## 6. PR / 合并节奏（MUST）

- **小步快合**：单 PR 改动 ≤ 500 行（除非新模块脚手架）；> 8 小时未合入的 PR → 当天先 rebase main。
- **合入顺序优先级**（多个 PR 待合时）：
  1. SSoT 文件改动 PR（先把规范定死，其他人才能照新规矩走）
  2. proposal 文件 PR（与 SSoT 配套）
  3. 模块字段对照表 PR（[PRD-MAPPING.md §2](../PRD-MAPPING.md) 新增）
  4. 业务代码 / SQL PR
  5. 测试 / E2E PR
  6. 文档纯改动 PR
- **rebase 而非 merge**：合入 main 前必须 `git rebase origin/main`，禁止 merge commit 污染主线（除非 PR 体量极大需要"merge with"语义，提前商量）。
- **单 PR 单模块**：一次 PR 改 2 个业务模块 → 拒收，拆成两个。
- **PR 描述必填字段**（沿用 [开发规范.md §4.3](../03-开发/开发规范.md) + 增补）：
  - 背景 / 改动点 / 自测情况 / 关联 / 风险
  - **新增**：本 PR 触及的 SSoT 文件清单（若有）
  - **新增**：本 PR 期间台账状态（链到 [在途任务.md](在途任务.md) 对应行）

## 7. 通信契约（SHOULD）

- **Commit message scope 精确**：业务模块改 → `feat(testdata): ...`；多模块改不要混 commit；纯协作规范改 → `docs(collab): ...`。
- **Commit 之间留可重排空间**：每个 commit 应当能独立 cherry-pick；不要让 commit N 强依赖 commit N-1 的未提交状态。
- **Session 启动主动报到**：Claude 启动后第一句话告知"我在 worktree X 接活 Y"，便于人类把它登记进台账。
- **长任务节点同步**：> 4h 的任务每 2h `git fetch` 看进度；被打断时主动写台账"暂停"。
- **跨 session 转交**：A session 中途要把活交给 B → 在台账 Owner 列写明转交、写清当前进度（哪个文件改到哪、下一步要做啥）。

## 8. 冲突处理（MUST）

| 类型 | 处理 |
|---|---|
| **代码合并冲突**（业务文件，不同行） | 直接 rebase 解决；解决后跑一遍本模块 `mvn install` / 前端 build 验证 |
| **同文件同行冲突**（代码） | **停下 AskUserQuestion**；不许 Claude "聪明地"合并 |
| **SSoT 冲突** | 停下；ping commit author；走 proposal 记录决策 |
| **Gate 实例冲突** | 已签字方版本胜出；后改者走"修订记录"追加，禁止覆盖 |
| **proposal 编号冲突** | 后到者 +1，并在自己 proposal 顶部备注"原计划编号 X，因冲突改 Y" |
| **业务模块代码冲突**（两个 session 各自实现了同一模块） | 走 §4 串行原则；后到者**作废**自己的实现或改写为"对前者的补丁"，禁止两份并行合入 |

**严禁的反模式**：
- ❌ `git push --force` 覆盖别人的工作
- ❌ rebase 时丢弃别人的 commit（看到 commit author 不是自己 → 必须保留）
- ❌ "我先 merge，你后面 rebase"作为单方决策
- ❌ Claude 见冲突自己挑一边合（必须 AskUserQuestion）

## 9. 不可触碰区 — 跨 session 边界（MUST）

承接 [.claude/rules.md §B](../.claude/rules.md)，并增加多 session 维度：

- 其他 session 的 worktree 目录（`.claude/worktrees/<other-name>/`）→ **不要 cd 进去**、不要 edit、不要 git 操作
- 其他 session 的分支 → 不要 `git push` / `git branch -D` / `git reset` / `git checkout -B` 覆盖
- 已签字的 `gate-checklists/instances/*` → 走"修订记录"追加，不覆盖
- 已 merged 的 proposal → 同上
- main 分支保护：不直接推；不强推；不 reset
- `.claude/settings.local.json` → 各人个人配置，禁止读其他 session 的本地配置去推断他人偏好

## 10. Claude session 行为（MUST）

Claude 在多 session 场景下的额外义务：

1. **启动即报到**：第一轮响应前必跑 `git worktree list && git status && git log --oneline -3`，看清自己在哪、和 main 偏离多少
2. **写规范前查 §L.2**：要改 `.claude/rules.md` / 模板文件 / `模块工作流.md` → 先开 proposal，不直接动文件
3. **见冲突即停**：所有 §8 表中"停下 AskUserQuestion"场景，**禁止**自作主张取舍
4. **每完成一个高影响 commit**（触及 SSoT / 新模块 / 改 schema），主动建议刷新 [在途任务.md](在途任务.md) 状态
5. **session 末沉淀**（Stop hook 已提示）：本规范本身的进化条目也按 [.claude/rules.md §L.1](../.claude/rules.md) 沉淀机制处理 — 发现规范矛盾走 proposal

## 12. 现状速查（开工对照表）

新 session 上手 1 分钟检查表：

```
□ git worktree list 知道自己在哪
□ git fetch --all --prune 同步远端
□ git log --oneline origin/main..HEAD 看我领先 main 多少
□ git log --oneline HEAD..origin/main 看 main 领先我多少（> 10 commit 先 rebase）
□ cat 99-跨阶段/在途任务.md 看其他 session 在干啥
□ 我的活在台账加一行 / 不与现有行重叠
□ 我要改的文件不在 §2 SSoT 表 / 或我已经开了 proposal
□ 我要改的业务模块没有第二个 session 在改（§4）
□ commit 前再次确认 scope 与模块 / 触及 SSoT 清单一致
□ PR body 含触及 SSoT 列表 + 台账行链接
```

### 13.2 阶段 Owner（沿用 [模块工作流.md](模块工作流.md) Phase 表）

| Phase | 主 Owner | 多 session 场景下的边界 |
|---|---|---|
| 01 立项 | 产品 PM | 同模块同时只允许 1 个立项 session |
| 02 设计 | 架构师 + DBA + 设计师 | 三个分支 worktree 各自并行，最后由架构师合 |
| 03 开发 | 开发 lead | §4 同模块串行严格执行 |
| 04 测试 | QA lead | E2E 套件并发跑 OK，bug 修复回 Phase 03 走串行 |
| 05 上线 | DevOps / oncall | **不允许两人同时点发布按钮** |
| 06 运营 | 产品 / 运营 lead | 数据看板 / 反馈分类可并行 |

### 13.3 "main 守门员"角色（新设）

无论 solo 或多人模式，必须有一个"main 守门员"角色：

- **唯一权力**：接受 PR 合 main 的最后一道 review + merge 操作
- **轮值**：solo 模式默认项目维护者；团队模式每周轮换（写进 [团队 OKR.md](团队%20OKR.md) "本周 main 守门员"字段）
- **义务**：合 PR 前再过一遍 §6 合入顺序 + §2 SSoT 锁；CI 红了不许合
- **拒收清单**：单 PR 含两个业务模块改动 / 缺 SSoT 触及清单 / 缺台账行链接 / commit 不规范 / 未 rebase

## 14. 同步节奏（SHOULD，含部分 MUST）

异步协作不等于"不同步"。三个固定节拍降低 drift 风险。

### 14.1 每日（异步站会替代，SHOULD）

每个活跃 session（Claude 或人类）开工第一件事 / 收工最后一件事各跑一次：

```bash
git fetch --all --prune
git log --oneline origin/main..HEAD   # 我领先
git log --oneline HEAD..origin/main   # main 领先我
cat 99-跨阶段/在途任务.md             # 其他 session 在干啥
```

**输出预期**：< 30 秒；若发现自己 > 24h 没 fetch / main 领先 ≥ 10 commit → 先 rebase 再继续。

人类可用 git alias 简化：

```bash
git config --global alias.standup '!git fetch --all --prune && git log --oneline -10 origin/main..HEAD && echo "---" && git log --oneline -10 HEAD..origin/main'
git standup
```

### 14.2 每周（MUST）

周五（或自定周末日）做一次 ≤ 30 分钟同步：

- 看 [在途任务.md](在途任务.md) "已完成"段，挪 7 天前的行进归档
- 看"暂停"段，决定是否恢复或放弃
- 跑 `reflect/YYYY-WW.md` 周报（待 Phase B `/reflect-weekly` skill 上线后自动；当前手动）
- 本周冲突 / 阻塞写进周报"observations"段

**solo 模式**：可压缩到 10 分钟自我对话；写一条进周报即可。

### 14.3 每月（MUST）

每月 1-3 日做一次月度反思：

- 跑月报 `reflect/YYYY-MM.md`（待 Phase B `/reflect-monthly` skill）

## 15. 冲突升级路径（MUST）

§8 "停下 AskUserQuestion" 是入口；如果停不下来，按下面阶梯升级。**不要跳级**。

| 级别 | 何时 | 谁来管 | 输出 | 时限 |
|---|---|---|---|---|
| **L0 自解** | 普通 rebase / 不同行冲突 / 自己一个人改坏了 | session 自身 | 解决 + commit 备注"冲突已解" | < 30 min |
| **L1 owner 协调** | 两 session 改同一文件不同行；台账撞行；同模块并行嫌疑 | 两 session 的 Owner 直接 ping | 一方让位 / 拆 commit / 改时序 | < 2 h |
| **L2 周报讨论** | L1 协调不下；规范本身模糊；同类冲突 1 周内 ≥ 3 次 | 项目维护者主持周会 | reflect 周报"observation" + 候选 proposal | ≤ 1 周 |
| **L3 proposal 决议** | 周报建议形成 / 需要改 SSoT 才能根治 | proposal 评审人（按 [proposals/README §评审节奏](proposals/README.md)） | merged proposal + SSoT 变更 | 按 proposal 类型时限 |
| **L4 管理升级** | proposal 评审分歧 / 涉及人事 / 项目方向问题 | 项目经理 + 业务负责人 | 决策纪要进 [会议纪要/](会议纪要/) | ≤ 1 周 |

### 15.1 严禁的"跳级"反模式

- ❌ L0 没尝试就 ping 别人（自己 rebase 一下能解的事不要打扰）
- ❌ L1 解了就完事不 reflect（同类冲突再现说明规范有 bug）
- ❌ 直接 L4 越级（管理只看决议级别问题）
- ❌ 把 L3 proposal 当吵架工具（评审会拍板，不是辩论）

### 15.2 升级触发标记

commit message / PR / 台账备注里统一标记：

| 标记 | 含义 |
|---|---|
| `[L1-coord]` | L1 协调过的改动 |
| `[L2-reflect-候选]` | 标记给周报关注 |
| `[L3-proposal-NNNN]` | 关联到 proposal NNNN |
| `[L4-mgr-decision]` | 管理决议落地 |

## 16. 新 Session 入退场流程（MUST）

### 16.1 入场清单（新 session 上手 5 分钟）

```
□ git worktree list                              # 我在哪、还有谁
□ git fetch --all --prune                        # 同步远端
□ git status / git log --oneline -5              # 我的分支偏离
□ cat CLAUDE.md                                  # 项目概览 + 4 个坑
□ cat .claude/rules.md                           # Claude 硬约束（自动加载）
□ cat 99-跨阶段/协作规范.md                       # 本规范（多 session 契约）
□ cat 99-跨阶段/在途任务.md                       # 其他 session 在干啥
□ ls 99-跨阶段/proposals/ | tail -3              # 最新 proposal 编号
□ git log --oneline -1 origin/main               # main 最新 commit
□ 在 在途任务.md 加自己的"进行中"行
```

Claude session 自动跑前 7 项（启动 hook 或主动报到）；人类按 README 走。

### 16.2 完工退场清单

```
□ 自测通过 (mvn install / npm run build / E2E 关键 case)
□ commit message 符合 Conventional Commits + 含 SSoT 触及清单
□ rebase 到 origin/main 最新
□ PR 描述含背景/改动/自测/风险 + 台账行链接
□ git push -u origin <branch>:<branch> 显式 refspec(gotchas §6)
□ PR 合入后,把 在途任务.md 该行从"进行中"挪到"已完成"
□ 若该 session 后续不再使用 → git worktree remove <path>
□ 已合入分支保留 7 天后由 main 守门员清理
```

### 16.3 转交流程（A → B）

A session 中途让 B 接手：

1. A 在 在途任务.md 的 "Owner" 列写 `A → B (2026-MM-DD 转交)`
2. A 在分支末尾追一个空 commit 写交接笔记：
   ```bash
   git commit --allow-empty -m "chore(handoff): A→B — 当前进度 X，下一步 Y，注意 Z"
   ```
3. A push；B fetch + checkout 该分支
4. B 接手第一动作：跑入场清单 §16.1 + 看交接 commit
5. 完工时 B 走退场清单 §16.2

### 16.4 worktree 清理周期

- 已合入 main 的 `claude/*` 分支 → 7 天后 main 守门员删（`git branch -d claude/<name> && git worktree remove .claude/worktrees/<name>`）
- "暂停" > 30 天的 worktree → 重新评估；放弃则归档分支（`git branch -m claude/<name> archive/claude-<name>` 后删 worktree）
- 远端死分支同步清理（`git push origin --delete claude/<name>`）

## 17. 失败回滚剧本（MUST）

按"破坏面"分级。**先回滚再说话**，复盘留给 reflect。

### 17.1 SSoT 文件改坏

**症状**：刚 merge 一个改 PRD-MAPPING.md / rules.md / 开发规范.md 的 PR 后，其他 session 大量 fetch 报错 / Claude 行为异常。

**回滚**：

```bash
git checkout main
git revert <bad-commit-sha> --no-edit
git push                       # main 守门员操作
```

**通知**：在 在途任务.md 顶部 placard 区写 "⚠ <文件> 已回滚至 <good-sha>，原因 <reason>，新一轮请 owner 重做"，并 ping 所有活跃 session。

**复盘**：reflect 周报记 "L2 候选" + 起 proposal 优化"SSoT 改动前 dry-run 检查"。

### 17.2 Proposal 走偏

**症状**：merged proposal 进 tracking 期后，监测指标恶化（违反原 §8 衡量指标）。

**回滚**：

1. 在该 proposal 的 §10 "实施后跟踪"段标记 `[ ] reverted`
2. 写"为什么失败"（数据 + 反思）
3. 起反向 proposal `NNNN-revert-XXXX-<原标题>.md`（编号递增），引用原 proposal
4. 反向 proposal merged 后，回滚 SSoT 变更

**复盘**：保留原 + 反向两份 proposal 在 git history。**禁止删 rejected/reverted proposal**（按 [.claude/rules.md §L.4](../.claude/rules.md) 数据完整性）。

### 17.3 Gate 实例误签

**症状**：已签字的 `gate-checklists/instances/<m>/PhaseNN-...-Gate-<date>.md` 后发现内容错（如把 P0 bug 漏过当成 ✅）。

**回滚**：**禁止**覆盖编辑该实例。新建"修订记录"附加章节：

```markdown
## 修订记录

| 日期 | 修订人 | 改了什么 |
|---|---|---|
| YYYY-MM-DD | <name> | §X.Y 项 ✅ → ❌；理由 <详细>；上线后果 <说明> |
```

若原 Gate 已经放行了下一阶段，**走 17.1 SSoT 回滚 + 回到该阶段重做**。

### 17.4 上线失败 / 数据脏

参 [05-上线/Runbook.md](../05-上线/Runbook.md) 该模块的"回滚步骤"段。一般序列：

1. 触发预设 rollback（容器版本 / DB migration / feature flag 关闭）
2. 通知 oncall + 在 在途任务.md 顶部 placard 区写 "🚨 P0 incident <时间> <module>"
3. 24h 内写 incident 复盘进 [06-运营/](../06-运营/) 或 reflect
4. 起 P0 hotfix 走 [模块工作流.md 分级](模块工作流.md)（跳 01/02 + 48h 内补齐）

### 17.5 worktree 误操作

| 误操作 | 回滚 |
|---|---|
| 在错误 worktree 改了文件 | `git stash` → 切到正确 worktree → `git stash pop` |
| `git reset --hard` 丢了 commit | `git reflog` 找回；30 天内可恢复 |
| 误 push 到他人分支 | 立即 ping 该分支 owner；**不要自行 force push 撤销** |
| 误 push 到 main | main 守门员走 17.1 revert；**勿 force push main** |

## 18. Claude-Claude 协作专项（MUST）

多个 Claude session 同时跑时（如 5 个 worktree 并行），Claude 之间的隐性冲突最难发现。下面是 Claude 视角额外规则。

### 18.1 礼让原则

- **见 origin/main 有比自己更新的 SSoT 改动** → 先 rebase 再继续，**不要**强推自己版本
- **看到 在途任务.md 已有人接同样的活** → 不要重复开工；改接相邻任务或写"我也对这块感兴趣，等你完工"
- **Plan mode 撞同 SSoT** → 先 ExitPlanMode 等对方 commit，再重新 Plan（避免两个 Plan 都基于"旧" SSoT）

### 18.2 资源争抢仲裁

下列资源在多 Claude 场景下必须独占：

| 资源 | 仲裁方式 |
|---|---|
| MySQL `plm` schema 写操作 | 用前先 `git fetch` 看是否有未合并的 schema 改动；冲突时按 §4 同模块串行 |
| 8081 端口（后端） / 80 端口（前端 dev） | 同机只能一个 Claude 跑；先到先得，后到的用 8082/81 等 |
| `.claude/skills/*/references/gotchas.md` | 多 Claude 都可能想加 gotcha；约定"加完先 fetch 看是否撞行" |
| `99-跨阶段/在途任务.md` | 每个 Claude 只动自己那行，撞行走 §8 冲突处理 |

### 18.3 交接笔记最低标准

A Claude → B Claude 转交时（§16.3），交接 commit message 必须含：

```
chore(handoff): <session-A> → <session-B> — <module>

当前进度:
- 已完成: <文件 1>, <文件 2>, ...
- 未完成: <下一步具体动作>
- 阻塞 / 风险: <若有>

环境/上下文:
- 后端运行在 :8081 / 前端 :80
- DB schema 版本: <commit-sha> / <SQL 文件>
- 关键决策: <ADR 链接 / proposal 链接 / 会话引用>

下一步 B 应当:
1. <具体动作>
2. <具体动作>

测试状态:
- mvn install: PASS / FAIL
- E2E: NN/MM passed
```

### 18.4 自批自审禁令（沿用 [.claude/rules.md § N.2](../.claude/rules.md)）

Claude **不允许**：

- 写一个 proposal 然后立刻自己 "solo-approve" + commit SSoT 改动
- 给自己加 permission 白名单（settings.local.json / .claude/rules.md 等）
- 用 AskUserQuestion 模糊问题诱导用户简单 "继续" 作为 SSoT 改动的具体授权

**正确路径**：proposal merged 必须经"用户明确选项点击 + 文字写明授权范围"两段；其中任何一段缺失 → 等待。

本规则由 2026-05-17 sleepy-hellman 会话亲身踩坑总结（proposal 0008 自身实施过程被 classifier 拦截 3 次 → 路径 A 用户加白名单后合规补回）。

### 18.5 多 Claude 元会话（meta-session）

如果用户在多个 Claude session 间提示词不一致（如 A session "用中文"、B session "always English"），不要互相覆盖。每个 Claude 在自己 worktree 内尊重用户**当前**给的指令；通过 commit message / 台账备注互相通气，不直接干预他人配置（`.claude/settings.local.json` 是个人配置，不读取/不推断他人偏好）。

### 18.6 hand-off vs spawn-task

Claude 在本会话内发现新 task 时，两条路径：

- **hand-off**：自己接着干，但生成 commit 让别的 session 后续 pickup（§16.3）
- **spawn-task**：调用 `mcp__ccd_session__spawn_task` 工具开新背景 session 并行处理（适合 out-of-scope 的小修复，主会话不要打断）

避免直接 `Agent` 工具开 subagent 处理"跨 session 协作"任务，subagent 不能 push commit，会造成"幽灵改动"。
