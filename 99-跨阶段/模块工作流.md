# 模块工作流（按 7 阶段生命周期）

每个业务模块（如 `项目 Project / 阶段 Phase / 任务 Task / 需求 Requirement / 变更 ChangeRequest / 评审 Review / 风险 Risk`）从立项到上线运营都按本表流转。

**核心原则**：每阶段都有"准入条件 → 必产出物 → 准出条件（Definition of Done）→ 进入下一阶段"。Owner 不签字、产出物没有，**不许进入下一阶段**。

---

## 🚦 强制执行层 — Gate Checklist

本文档**只是描述**。真正的卡控发生在 [gate-checklists/](gate-checklists/) 目录：

| 阶段 | 描述（本文件）| 执行模板（强制） |
|---|---|---|
| 01 立项 | 见 Phase 01 段 | [gate-checklists/Phase01-立项-Gate.md](gate-checklists/Phase01-立项-Gate.md) |
| 02 设计 | 见 Phase 02 段 | [gate-checklists/Phase02-设计-Gate.md](gate-checklists/Phase02-设计-Gate.md) |
| 03 开发 | 见 Phase 03 段 | [gate-checklists/Phase03-开发-Gate.md](gate-checklists/Phase03-开发-Gate.md) |
| 04 测试 | 见 Phase 04 段 | [gate-checklists/Phase04-测试-Gate.md](gate-checklists/Phase04-测试-Gate.md) |
| 05 上线 | 见 Phase 05 段 | [gate-checklists/Phase05-上线-Gate.md](gate-checklists/Phase05-上线-Gate.md) |
| 06 运营 | 见 Phase 06 段 | [gate-checklists/Phase06-运营-Gate.md](gate-checklists/Phase06-运营-Gate.md) |

**操作流程**：每个业务模块、每个阶段都复制对应模板到 `gate-checklists/instances/<模块>/`，打勾签字，commit 入库。**不打勾 = 不签字 = 不许进入下一阶段**。

**分级走**（详见 [gate-checklists/README.md](gate-checklists/README.md)）：
- **L1 重大**（新业务模块 / 主架构变更 / DB 结构性变更 / 安全合规）→ 走全部 6 个 Gate
- **L2 中型**（已有模块大改 / 新增 feature / 新增公开 API）→ 走 02-06，01 可简化
- **L3 小型**（bug / typo / 小重构 / patch 升级）→ 仅 PR + CR + 04 回归子集 + 05 上线
- **P0 hotfix** → 跳 01/02 但 48h 内补齐

```
01-立项  →  02-设计  →  03-开发  →  04-测试  →  05-上线  →  06-运营
```

## Phase 01 — 立项

| 项 | 内容 |
|---|---|
| **准入条件** | 有明确的业务痛点 / 机会点，初步价值假设 |
| **必产出物（按顺序）** | 1. [01-立项/市场调研.md](../01-立项/市场调研.md) → 2. [01-立项/PRD.md](../01-立项/PRD.md) → 3. [01-立项/商业计划.md](../01-立项/商业计划.md) → 4. [01-立项/立项评审纪要.md](../01-立项/立项评审纪要.md) |
| **Owner** | 产品经理 / 业务负责人 |
| **参与方** | 业务、技术 lead、设计 lead、（必要时）法务、合规、CFO |
| **准出条件（DoD）** | ✅ PRD v1.0 已评审 ✅ 评审纪要含决议（通过 / 有条件通过 / 不通过）✅ 风险已登记到 [99-跨阶段/风险登记册.md](风险登记册.md) ✅ 排上 OKR（[99-跨阶段/团队 OKR.md](团队%20OKR.md)） |
| **不允许跳过** | PRD（无 PRD 不开发）、立项评审纪要（无评审不进入下阶段） |
| **典型耗时** | 1–4 周 |

## Phase 03 — 开发

| 项 | 内容 |
|---|---|
| **准入条件** | Phase 02 全部 DoD 满足；API 契约和 DB 设计已冻结 |
| **必产出物** | 1. 业务代码（按 [开发规范.md](../03-开发/开发规范.md) 落位）2. SQL 迁移脚本（`plm-backend/sql/business-<entity>.sql`）3. 单元 / 集成测试代码 4. Sprint 计划与回顾纪要 → [03-开发/Sprint 计划与回顾/](../03-开发/Sprint%20计划与回顾/) 5. 必要的 ADR → [03-开发/ADR/](../03-开发/ADR/) |
| **Owner** | 开发 lead |
| **参与方** | 后端、前端、QA（参与早期联调） |
| **进入下一阶段的 DoD** | ✅ 代码合 main 且 CI 全绿 ✅ 单元测试覆盖率达标（Service ≥ 70%）✅ 关键路径自测通过 ✅ Swagger 文档与实现一致 ✅ DB 脚本在 staging 已演练 ✅ Sprint 回顾纪要已归档 |

| **不允许跳过** | 单元测试（Service / 复杂 utils）、Sprint 回顾（哪怕一句话） |

## 卡控节点（Gate Review）

下列时机必须开一次正式评审，纪要进 [99-跨阶段/会议纪要/](会议纪要/)：

1. **立项评审**（Phase 01 → 02）：业务、技术、CFO 共同拍板
2. **设计评审**（Phase 02 → 03）：架构 + DBA + 测试 + 安全共同审
3. **代码冻结评审**（Phase 03 → 04）：开发 + 测试确认范围
4. **发布评审**（Phase 04 → 05）：所有 oncall 在场，确认回滚方案
5. **季度运营复盘**（Phase 06 每季度）：对照 OKR 看是否符合预期

## Definition of Done — 通用判据

任意阶段产出物，"完成"必须满足以下全部：

- [ ] 文档落在 [上述目录结构](../) 的正确位置（按 `0?-阶段名/` 归类）
- [ ] 文件命名符合 [开发规范.md §6.2](../03-开发/开发规范.md)
- [ ] 内容包含必要小节（不是空模板）
- [ ] Owner 字段填写明确（人 + 角色），不能只写"我们组"
- [ ] 已在评审会议上过一遍，会议纪要归档
- [ ] 关联的风险已录入 [风险登记册.md](风险登记册.md)

## 多 Session 协作

模块按本工作流走 6 阶段；当多个 session（Claude / 人 / 人机混合）同时工作时，额外遵守 [协作规范.md](协作规范.md)：

- **同模块串行**：任一业务模块同一时刻只允许一个 session 改代码 + SQL，避免两份不一致实现（[协作规范.md §4](协作规范.md)）
- **Gate 实例签字**：同模块同阶段同一天的 Gate 实例文件名加 `-v2` 后缀保留双签，禁止覆盖（[协作规范.md §5](协作规范.md)）
- **PR 合入顺序**：SSoT 文件改动 > proposal > 字段对照表 > 业务代码 > 测试 > 文档（[协作规范.md §6](协作规范.md)）
- **任务可见**：每个并行任务在 [在途任务.md](在途任务.md) 登记一行，开工/转场/收工各自维护

详尽契约（worktree 隔离 / SSoT 资源锁 / 冲突处理 / Claude session 行为）见 [协作规范.md](协作规范.md)；Claude 硬条款摘要见 [.claude/rules.md § N](../.claude/rules.md)。

## 反模式（拒绝执行）

- ❌ 没有 PRD 就开始写代码
- ❌ 数据库设计边写边改（导致 sql 脚本和实际表结构对不上）
- ❌ Sprint 末才发现 API 契约和实现不一致
- ❌ 上线前才补测试
- ❌ 没有回滚方案就发布
- ❌ 运营发现的反馈不立 issue 直接私聊开发
- ❌ ADR 写完不评审就标 accepted

发现这些反模式，记入下次 Sprint 回顾的"可改进"。
