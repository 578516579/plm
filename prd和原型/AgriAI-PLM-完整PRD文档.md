# 一、竞品调研报告

## 1.1 市场背景

2025年AI原生研发工具进入爆发期。Forrester预测30%的团队将通过TuringBots加速SDLC；Gartner预测到2028年33%企业软件将包含Agentic AI。当前市场呈现"传统项目管理工具 + AI补丁"与"AI原生研发平台"两大阵营的分化格局。

对农业信息化行业而言，现有通用工具无法满足：农业业务周期强季节性、实施场景下沉（弱网/离线）、行业术语与知识体系特殊等核心诉求。

## 1.2 主要竞品分析

### 竞品一：禅道（ZenTao）
- **定位**: 国产开源研发项目管理，覆盖需求→任务→Bug全流程
- **AI能力**: 浅层，主要在代码评审、文档生成辅助，尚未形成AI驱动的流程编排
- **优势**: 国产化、本地部署、IPD流程支持、价格亲民
- **劣势**: AI能力弱、无Dify/MCP集成、无农业垂直知识库、自动化测试能力薄弱
- **价格**: 社区版免费，企业版约2万/年起
- **启示**: 本土化方向正确，但AI深度不足，我们需要在此方向上"降维打击"

### 竞品二：LigaAI
- **定位**: 智能研发管理平台，强调AI自动化流转
- **AI能力**: 自动进度填报、智能指派、数据可视化，AI融合度较高
- **优势**: 研发流程自动化、三方集成能力、异步协作
- **劣势**: 无农业行业知识、无全生命周期覆盖（缺测试、文档体系）、无MCP生态
- **价格**: 按席位，约400元/人/月
- **启示**: AI驱动流转是正确路径，需延伸至测试、文档、实施全链路

### 竞品三：Atlassian Jira + Rovo Dev
- **定位**: 全球最大研发协作平台 + AI研发助手
- **AI能力**: Teamwork Graph跨项目知识连接、Rovo Dev CLI集成Jira/Bitbucket/SDLC
- **优势**: 生态完整、MCP/OAuth集成、企业级治理
- **劣势**: 价格高（企业版数万元/月）、国内访问慢、无中文农业场景适配、私有化部署困难
- **价格**: Jira Premium约$17.5/用户/月
- **启示**: MCP+CLI的整合方向是标准，全链路打通是核心竞争力

### 竞品四：GitHub Copilot Workspace
- **定位**: AI原生编码工作空间，从Issue到PR全自动
- **AI能力**: 多模型支持（GPT/Claude/Gemini）、Agent模式多文件编辑、MCP OAuth
- **优势**: 代码生成最强、生态最广、模型选择灵活
- **劣势**: 仅覆盖编码环节、无立项/需求/测试/文档管理、不适合中国私有化
- **价格**: Enterprise $39/用户/月
- **启示**: 编码端AI是标配，差异化在于上下游链路打通

### 竞品五：Cursor / Claude Code（AI IDE）
- **定位**: AI原生IDE，Vibe Coding代表
- **AI能力**: 多文件Composer、上下文理解、Agent自主执行
- **优势**: 编码效率提升35~55%、MCP生态丰富
- **劣势**: 仅是IDE工具，无项目管理、需求、测试等上层能力
- **价格**: $20/月/人
- **启示**: 作为系统研发环节的内嵌工具，通过MCP接入我们的平台

### 竞品六：PingCode（WorkItem）
- **定位**: 研发管理SaaS，面向中国市场
- **AI能力**: 有限的AI辅助（需求拆解、报告生成），尚在探索期
- **优势**: 本土化好、集成飞书/钉钉、研发流程完整
- **劣势**: AI深度浅、无Dify编排能力、农业垂直化缺失
- **启示**: 本土SaaS集成路径可参考，Dify编排是我们的差异化壁垒

## 1.4 市场机会总结

# 二、产品方案

## 2.1 产品定位

**AgriPLM·AI** —— 面向农业信息化行业产研团队的 **AI原生项目全生命周期管理平台**

核心价值主张：
> "让AI成为每个产研角色的超级助手，从一个想法到一套完整的产品，全程AI协同、自动流转、知识沉淀。"

## 2.2 核心架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                    AgriPLM·AI 平台层                        │
├─────────────┬──────────────┬──────────────┬─────────────────┤
│  立项决策域  │  产品设计域  │   研发执行域  │   质量保障域   │
│  Project    │  Product     │  Dev         │   QA            │
│  Inception  │  Design      │  Execution   │   Assurance     │
├─────────────┴──────────────┴──────────────┴─────────────────┤
│                    AI 编排引擎层（Dify）                     │
│   工作流编排 | Agent调度 | 知识库RAG | 多模型路由            │
├─────────────────────────────────────────────────────────────┤
│                    集成适配层                               │
│   MCP Server | CLI工具 | Webhook | REST API | 消息队列       │
├──────────┬───────────┬───────────┬──────────┬───────────────┤
│  Jira    │  GitLab   │  飞书/钉钉 │  Figma  │  禅道/ZTF     │
│  接入    │  接入     │  接入      │  接入   │  迁移/对接     │
└──────────┴───────────┴───────────┴──────────┴───────────────┘
```

## 2.3 AI能力矩阵（基于Dify编排）

每个阶段均通过Dify流程编排，调用大模型完成核心任务：

| 阶段 | Dify工作流名称 | 核心AI能力 | 输出物 |
|------|--------------|-----------|--------|

| 需求设计 | requirements-flow | 用户故事生成、优先级评分 | 需求规格书 |
| PRD生成 | prd-generation-flow | 结构化PRD自动生成 | PRD文档 |
| UED设计 | ued-review-flow | 设计规范检查、可用性建议 | 设计评审报告 |
| 概要设计 | arch-design-flow | 架构推荐、技术选型建议 | HLD文档 |
| 数据库设计 | db-design-flow | ER图生成、规范检查 | 数据库设计文档 |
| 详细设计 | detail-design-flow | 接口设计、类图生成 | LLD文档 |
| 编码辅助 | coding-assist-flow | MCP调用Claude Code/Cursor | 代码+注释 |
| 单元测试 | unit-test-flow | 测试用例自动生成 | 测试代码 |
| 测试用例 | testcase-gen-flow | 基于需求+边界分析生成用例 | 测试用例集 |
| 测试方案 | test-plan-flow | 测试策略、范围、工具选型 | 测试方案书 |
| 测试报告 | test-report-flow | 缺陷统计、质量评估、建议 | 测试报告 |
| API文档 | api-doc-flow | 从代码自动生成OpenAPI文档 | API文档 |
| 造数 | data-gen-flow | 根据字段语义生成测试数据 | 测试数据集 |
| 产品手册 | product-manual-flow | 功能+截图自动生成手册 | 产品手册 |
| 实施手册 | impl-manual-flow | 部署步骤+配置生成 | 实施手册 |
| 运维手册 | ops-manual-flow | 监控+告警+排障生成 | 运维手册 |

## 2.4 农业行业知识库设计

**AgriKnowledge Base（AgriKB）**包含：
- 农业信息化行业标准（NY/T系列、GB/T农业标准）
- 常见农业业务术语（种植管理、农资流通、溯源体系、植保服务）
- 历史项目积累的需求模式库、Bug模式库
- 农业场景测试数据集（土壤传感器、气象站、无人机等IoT数据）

## 2.5 MCP生态集成设计

```
AgriPLM MCP Server 暴露的工具集:
├── project_tools: 立项查询/创建/更新
├── requirement_tools: 需求CRUD/关联
├── task_tools: 任务分派/进度更新
├── testcase_tools: 用例生成/执行/结果

└── data_tools: 测试数据生成/管理

├── agriplm init <project>  # 初始化项目
├── agriplm generate prd    # AI生成PRD
├── agriplm generate testcase --req <id>  # 生成测试用例
├── agriplm run test        # 执行自动化测试
├── agriplm generate doc --type manual   # 生成手册
└── agriplm sync --target jira/zentao    # 同步到外部系统
```

# 三、产品需求文档（PRD）

## 3.1 产品概述

### 产品名称
AgriPLM·AI（农业信息化AI项目全生命周期管理平台）

### 产品愿景
成为农业信息化行业产研团队的"AI协作大脑"，将人均项目交付效率提升50%，文档生成时间减少80%，测试覆盖率提升至90%+。

### 目标用户

| 角色 | 核心诉求 | 使用频率 |
|------|---------|---------|
| 产品经理 | 快速生成PRD、竞品分析、需求评审 | 日常高频 |
| UED设计师 | 设计规范检查、标注生成、可用性报告 | 项目周期内 |
| 研发工程师 | 设计文档参考、代码辅助、API文档 | 日常高频 |
| 测试工程师 | 测试用例生成、造数、自动化执行 | 日常高频 |
| 项目经理 | 进度监控、风险预警、报告生成 | 日常高频 |
| 实施工程师 | 实施手册、部署文档、培训材料 | 项目实施期 |
| 部门负责人 | 效能分析、资源规划、质量看板 | 周/月 |

### 产品边界
- **包含**: 从立项到运维手册的全生命周期管理
- **不包含**: 财务管理、HR管理、客户CRM（通过MCP与外部系统对接）
- **集成**: 通过MCP/CLI与Jira、GitLab、飞书、钉钉、Figma、禅道、ZTF等系统双向同步

## 3.2 功能模块详细需求

### 模块1：项目立项管理（Project Inception）

**功能概述**: AI辅助完成从项目想法到立项报告的全流程

**核心功能**:

F1.1 **AI立项助手**
- 输入：项目名称、背景描述（自然语言，支持语音输入）
- AI处理：自动分析市场背景、预估工作量、识别风险点、生成ROI分析
- 输出：结构化立项建议书（可编辑、可导出Word/PDF）
- 支持引用农业行业知识库中的行业报告数据

F1.2 **项目档案管理**
- 项目基础信息（名称、类型、优先级、业务线）
- 项目阶段甘特图（支持拖拽调整）
- 人员组织架构（角色分配、权限管理）
- 项目健康度仪表盘（进度/质量/风险三色预警）

F1.3 **竞品情报系统**
- AI自动爬取并分析竞品官网、App Store评论、行业报告
- 支持手动上传竞品截图，AI解析功能特性
- 输出竞品对比矩阵（可自定义维度）
- 竞品动态订阅（关键词监控，变更自动推送）

**验收标准**:
- 立项报告AI生成时间 < 3分钟
- 竞品分析报告覆盖 ≥ 5个竞品维度
- 立项流程审批节点支持钉钉/飞书消息通知

---

### 模块2：需求与产品设计（Product Design）

**功能概述**: AI辅助需求采集、分析、PRD生成、UED协同

**核心功能**:

F2.1 **需求采集与管理**
- 需求来源多渠道接入（用户反馈、客户需求单、内部提案）
- AI自动归类、去重、优先级初评
- 需求关联图谱（父子关联、冲突识别）
- 需求状态机：草稿→评审→确认→开发中→完成→验收

F2.2 **AI PRD生成器**
- 输入：需求描述（自然语言）+ 选择业务场景模板
- AI生成标准PRD：
  - 背景与目标
  - 用户故事（User Stories）
  - 功能描述（含异常流程）
  - 非功能需求（性能、安全、兼容性）
  - 验收标准
  - 原型说明（文字描述原型交互）
- 支持多轮对话精炼PRD内容
- PRD版本管理与变更对比

F2.3 **UED设计协同**
- 与Figma MCP集成：自动同步设计稿版本
- AI设计评审：检查设计规范遵从度、可用性问题
- 标注自动生成（间距、颜色、字体规范）
- 设计与需求双向关联（需求变更自动提醒设计更新）
- 农业场景UI组件库（农情大屏、移动端农事记录、IoT数据展示等）

F2.4 **需求评审管理**
- 在线评审会议（集成腾讯会议/飞书会议链接）
- AI生成评审意见摘要
- 评审结论自动更新需求状态

**验收标准**:
- PRD AI生成完整度 ≥ 80%（减少人工填写量）
- 需求追踪矩阵自动维护，覆盖率100%
- UED与需求双向关联准确率 ≥ 95%

### 模块3：研发执行管理（Dev Execution）

**功能概述**: 覆盖架构设计→数据库设计→详细设计→编码→单测→联调全流程

**核心功能**:

F3.1 **系统概要设计（HLD）**
- AI根据PRD自动推荐技术架构方案（微服务/单体/混合）
- 生成系统架构图（C4模型：上下文图、容器图）
- 技术选型建议（附带优劣分析）
- 非功能需求映射到架构决策

F3.2 **数据库设计**
- AI根据业务实体自动生成ER图
- 建表SQL生成与规范检查（命名规范、索引建议、范式检查）
- 数据字典自动生成
- 支持导入已有数据库结构进行分析优化

F3.3 **详细设计（LLD）**
- API接口设计（OpenAPI 3.0规范自动生成）
- 类图/时序图AI辅助生成（Mermaid格式）
- 核心算法逻辑描述
- 联动Dify调用Claude Code生成代码框架

F3.4 **研发任务管理**
- 需求自动拆分为开发任务（支持AI估算工时）
- Kanban看板（待开发/开发中/代码评审/测试/完成）
- 代码提交关联任务（GitLab MCP集成）
- 研发日报AI生成（根据任务完成情况）

F3.5 **AI编码辅助集成**
- 通过MCP集成Claude Code/Cursor/GitHub Copilot
- 在设计文档中直接触发"生成代码框架"
- 单元测试代码AI自动生成
- Code Review AI辅助（规范检查、安全扫描）

F3.6 **联调管理**
- 联调环境管理（开发/测试/预发）
- 接口Mock服务（根据OpenAPI自动生成）
- 联调问题跟踪与解决

**验收标准**:
- 从PRD到任务分解自动化率 ≥ 70%
- AI生成单元测试覆盖率 ≥ 60%
- 代码提交与任务关联率 ≥ 90%

---

### 模块4：质量保障管理（QA Assurance）

**功能概述**: 覆盖测试方案→测试用例→造数→提测→执行→自动化→报告全链路

**核心功能**:

F4.1 **测试计划与方案**
- AI根据需求规模、迭代周期自动生成测试计划
- 测试方案：范围界定、策略选择、工具推荐、风险评估
- 测试资源分配建议（人员/时间/环境）

F4.2 **AI测试用例生成**
- 输入：需求文档/PRD/接口文档
- AI生成：
  - 功能测试用例（含等价类、边界值、判定表）
  - 异常场景用例（网络中断、数据为空、权限不足）
  - 性能测试用例
  - 农业业务场景专项用例（离线同步、弱网重试、传感器数据异常等）
- 用例评审与确认流程
- 用例复用（历史用例推荐）

F4.3 **测试数据工厂（造数）**
- 根据数据字典和字段语义自动生成测试数据
- 支持农业特定数据（地块坐标、作物品种、农药信息、气象数据）
- 数据集版本管理
- 一键重置测试环境数据

F4.4 **提测管理**
- 提测单标准化（含提测范围、环境说明、风险提示）
- AI检查提测质量门禁（单测覆盖率、代码扫描通过、文档完整性）
- 提测审批流（测试经理审核）
- 提测未通过自动退回并说明原因

F4.5 **自动化测试管理**
- 测试脚本管理（Selenium/Playwright/接口自动化）
- AI辅助生成自动化测试脚本
- 测试执行调度（定时执行/提交触发/手动执行）
- 执行结果分析（失败截图、日志分析、AI根因分析）

F4.6 **缺陷管理**
- 缺陷生命周期（发现→确认→修复→验证→关闭）

## 4.2 核心页面交互描述

### 页面1：工作台（Dashboard）
**布局**: 三栏布局
- 左侧：导航侧边栏（图标+文字，可折叠）
- 中间主区：
  - 顶部：欢迎语 + AI助手输入框（支持"帮我生成XX的PRD"自然语言指令）
  - 中间：我的待办卡片（按紧急度排序，AI标注风险项）
  - 下方：在办项目进度卡片（进度条+健康度色块）
- 右侧：AI助手侧边栏（悬浮，随时唤起）

**AI助手交互**:
1. 用户输入："帮我生成农业气象预警功能的PRD"
2. AI询问补充信息：目标用户是谁？有哪些约束条件？
3. AI生成PRD草稿，在侧边栏预览
4. 用户确认后，自动创建PRD文档并关联至对应项目

### 页面2：AI PRD编辑器
**布局**: 双栏
- 左侧：PRD大纲树（章节导航）
- 右侧：富文本编辑区
  - 每个章节右上角有"AI重写"/"AI扩写"/"AI精简"按钮
  - 选中文本可右键呼出AI菜单
  - 底部工具栏：AI检查（完整性/歧义性/可测试性）

**特色交互**:
- 需求变更时，AI自动高亮受影响的章节
- 需求与测试用例双向联动（点击需求条目，侧栏展示关联用例）

### 页面3：测试用例生成
**交互流程**:
1. 选择需求/接口文档作为输入
2. 配置生成参数（测试类型、覆盖深度、优先级偏好）
3. 点击"AI生成用例"，实时流式展示生成过程
4. 批量确认/拒绝/编辑用例
5. 一键导入测试执行计划

### 页面4：测试数据工厂
**交互**:
- 选择数据库表或字段配置
- AI识别字段语义（"crop_name"→作物品种，自动从农业知识库取值）
- 支持设置数据规则（如地块坐标必须在中国境内）
- 一键生成并预览，支持导出CSV/JSON/SQL

### 页面5：AI文档生成器
**交互**:
- 选择文档类型（产品手册/实施手册/运维手册）
- 配置文档范围（选择项目模块）
- 可上传截图，AI自动识别并生成对应说明文字
- 流式生成，实时预览
- 导出多种格式（Word/PDF/Markdown/在线链接）

# 五、关键差异化总结

1. **Dify深度编排**: 不是AI点缀，而是AI驱动每个阶段的核心流程
2. **农业垂直知识库**: 独有的AgriKB，让AI生成的内容贴合农业业务实际
3. **全链路打通**: 从立项到运维手册，无需切换工具
4. **MCP+CLI生态**: 与行业主流工具无缝集成，降低迁移成本
5. **私有化部署**: 满足农业客户数据安全合规要求
6. **离线支持**: 适应农村/农场等弱网场景的实施环境
